## Context

v7.75 "Provenance" adds cryptographic substitution defense to the SDK's two threshold-secret subsystems, universalizes DST discipline across every internal identifier, and locks in the mutation-audit methodology as a permanent regression gate. The prior draft ADR documenting this release tried to be both the cryptographic specification and the decision record. That dual role produced drift: every implementation refinement required an ADR edit, and the document went stale between revisions. This ADR replaces it with a decision-record-only scope. The specification lives in source where the compiler and the test suite can verify it remains coherent.

v7.75 is also scoped to be a complete SDK release on its own. The prior draft referenced a "Phase D" for operator-layer publication of commitment entries; that scope has been folded into Phase C. The SDK ships the commitment-entry lifecycle in full â€” structs, serializers, verifiers, builders, schema registration, and atomic emission from the lifecycle functions. Operator deployments consume this surface without requiring SDK extensions. Every path through `GrantArtifactAccess`, `ProvisionSingleLog`, and `StoreMapping` that produces shares or KFrags also produces and attaches the corresponding commitment entry in the same atomic operation.

## Decision

v7.75 comprises six locked commitments. Each is specified in source; this ADR records the decision and the boundary.

### 1. Pedersen VSS replaces plain Shamir in both threshold-secret subsystems.

Escrow splits (`crypto/escrow/vss_v2.go`) and Umbral PRE grants (`crypto/artifact/pre.go`) both publish commitment sets alongside the shares and KFrags they emit. Share and CFrag verification require polynomial consistency against the committed polynomial in addition to any pre-existing checks. The construction, generator derivation, Fiat-Shamir transcript, wire formats, and test vectors are specification-in-source under `core/vss/` and `crypto/artifact/`.

The honest-dealer assumption is scoped to SDK-controlled dealers only: `ProvisionSingleLog`, `StoreMapping`, `grantUmbralPRE`. Client-generated-key scenarios (where a third party could produce shares inconsistent with a published commitment) are deferred to a future release that adds NIZK proofs of polynomial correctness or multi-party dealer protocols. Every SDK function that invokes VSS documents in its godoc which party is the dealer in the function's context and what failure mode occurs if the dealer is malicious.

Independent blinding polynomials per VSS event are structurally enforced by the `core/vss/pedersen` package API. The `Split` function accepts no external polynomial input; it generates fresh randomness via `crypto/rand` on every call. No polynomial coefficients are shared between events, between subsystems, or across time.

### 2. Universal length-prefix rule for SDK-internal DSTs.

Every SDK-internal cryptographic identifier routes through `crypto/hash.LengthPrefixed(dst, fields...)`. The helper writes a 2-byte big-endian length for the DST and for every field before hashing, preventing boundary-shift attacks across variable-length inputs. Raw-concatenation constructions for SplitIDs, commitment hashes, and domain-app identifiers are forbidden; any code that hashes variable-length fields must either route through `LengthPrefixed` or document an explicit exception.

The sole carveout is BLS12-381 hash-to-curve, which follows RFC 9380 per IETF specification. The cosignature domain tag and Proof-of-Possession domain tag used in `crypto/signatures/bls_verifier.go` follow the IETF hash-to-curve format (raw byte concatenation with a trailing length byte, per the RFC) and do NOT route through `LengthPrefixed`. The carveout is documented at the top of `bls_verifier.go` with the specific function names excluded and the reason (IETF interop with external BLS implementations).

Pre-v7.75 SplitID constructions that used raw concatenation are migrated as part of this release. The legacy `EscrowSplitDST` constant is deleted. The canonical SplitID constructions under this rule are:

- **Escrow SplitID:** `LengthPrefixed("ORTHOLOG-V7.75-ESCROW-SPLIT", []byte(dealerDID), nonce[:])` â€” where `dealerDID` is NFC-normalized at the caller boundary and `nonce` is a 32-byte value from `crypto/rand`. The length-prefixing of the nonce is uniform with the rule; the marginal byte cost is trivial and avoids case-by-case reasoning about "truly fixed" fields.

- **PRE Grant SplitID:** `LengthPrefixed("ORTHOLOG-V7.75-PRE-GRANT-SPLIT-ID-v1", []byte(grantorDID), []byte(recipientDID), artifactCID.Bytes())` â€” where both DIDs are NFC-normalized at the caller boundary and `artifactCID.Bytes()` is the 1-byte algorithm tag plus the variable-length digest, NEVER `artifactCID.Digest` alone. The `.Bytes()` form is load-bearing for cross-algorithm collision resistance: `storage.CID` supports algorithm agility via `RegisterAlgorithm`, and hashing only the digest would allow two CIDs with identical digests under different algorithm tags to produce colliding SplitIDs.

- **DLEQ challenge transcript:** `LengthPrefixed("ORTHOLOG-V7.75-DLEQ-CHALLENGE-v1", ...transcript fields...)` â€” the full transcript specification lives in `core/vss/transcript.go` godoc and the locked golden vector at `core/vss/testdata/transcript_vector.json`.

The versioned DST suffix (`-v1`) on the PRE Grant SplitID and the DLEQ challenge DST exists so that future migrations (e.g., a post-quantum commitment scheme that changes the underlying construction) can rev the version without breaking the rule. The escrow DST is unversioned because its input domain is stable: `(dealerDID, nonce)` is a minimal tuple that has no obvious evolution path.

### 3. On-log commitment schemas, published atomically with share/grant emission.

Two new schemas, `escrow-split-commitment-v1` and `pre-grant-commitment-v1`, carry the commitment set as Path A commentary entries. They are published atomically with the share or grant emission that references them. Atomicity is structural in the lifecycle API: `grantUmbralPRE`, `ProvisionSingleLog`, and `StoreMapping` emit the commitment entry in the same log batch as the operation that produces the shares or KFrags. Callers cannot legally emit a grant without the commitment entry; the SDK function returns both or neither. An escrow node or recipient receiving a share or KFrag without a corresponding on-log commitment entry (verified via log query) MUST reject the material as structurally malformed.

The PRE grant SplitID is deterministic from the `(grantorDID, recipientDID, artifactCID)` tuple so recipients and proxies can locate the commitment entry from public grant context without out-of-band discovery. The deterministic derivation means the dealer cannot produce two distinct commitment entries for the same `(grantor, recipient, artifact)` tuple: the second attempt is either rejected at admission by schema uniqueness or treated as cryptographic evidence of equivocation by verifiers. Genuine re-granting of access to the same artifact requires rotating the `artifactCID`: revoke the old grant on-log, re-encrypt the artifact via `ReEncryptWithGrant` to produce a new CID, then issue a new grant against the new CID. This forces revocation and re-grant to be observable on-log rather than silent.

The escrow SplitID carries a random 32-byte nonce so legitimate re-splitting of the same master secret (during delegation-key rotation, for example) produces a fresh SplitID and a fresh commitment entry. The nonce is recorded alongside the commitment entry for audit.

Verifier-side equivocation detection per Property 7 of the security analysis: if a malicious dealer publishes two commitment entries for the same SplitID, both entries exist on the log under the dealer's signature. Verifiers encountering two matching entries MUST reject all shares associated with either commitment set, record the equivocation evidence (both entries, their sequence numbers, their witness cosignatures) for domain-level governance action, and refuse any reconstruction attempt. Honest dealers produce exactly one commitment entry per SplitID; equivocation is evidence of malice and is handled as such by the lifecycle layer.

### 4. Complete commitment-entry surface within the SDK.

v7.75 ships the full commitment-entry lifecycle. No part of this surface is deferred to a downstream phase or to operator-specific extensions.

- **Structs.** `PREGrantCommitment` in `crypto/artifact/pre_grant_commitment.go` and `EscrowSplitCommitment` in `crypto/escrow/split_commitment.go`, both carrying `SplitID`, threshold parameters, and the compressed-point commitment set. In-memory `vss.Commitments` uses 65-byte uncompressed points; the wire structs use 33-byte compressed points. The RAM-vs-wire asymmetry is documented in the struct godoc.

- **Serializers.** `SerializePREGrantCommitment` / `DeserializePREGrantCommitment` and `SerializeEscrowSplitCommitment` / `DeserializeEscrowSplitCommitment`. Deserialization validates size equals the M-derived expected length, threshold bounds `2 <= M <= N <= 255`, and every point on-curve.

- **Verifiers.** `VerifyPREGrantCommitment(c, grantorDID, recipientDID, artifactCID)` and `VerifyEscrowSplitCommitment(c, dealerDID, nonce)`. Each verifies exactly four properties: every commitment point on-curve, commitment set length matches M, threshold bounds, and SplitID recomputes to the expected value. Each has four mutation switches gating the four checks. Neither verifies envelope signatures, log membership, or caller authorization; those are lifecycle-layer concerns handled separately.

- **Builders.** `BuildPREGrantCommitmentEntry` and `BuildEscrowSplitCommitmentEntry` in `builder/entry_builders.go`. Each takes the commitment struct, signing key, and destination, producing a signed Path A commentary entry ready for admission. The builders follow the existing `validateCommon` destination-binding discipline.

- **Schema registration.** `schema/pre_grant_commitment_v1.go` and `schema/escrow_split_commitment_v1.go` declare the schemas so the admission pipeline validates entries at receive time. Both schemas extend `parameters_json_roundtrip_test.go` coverage.

- **Atomic lifecycle emission.** `grantUmbralPRE` emits the `pre-grant-commitment-v1` entry in the same batch as the grant. `ProvisionSingleLog` and `StoreMapping` emit the `escrow-split-commitment-v1` entry in the same batch as the split. The atomicity invariant is enforced by the function signatures: lifecycle functions return the commitment entry alongside the shares/KFrags, and the return types are structured such that a caller cannot extract one without the other.

- **Lookup primitives.** `FetchPREGrantCommitment(fetcher, grantorDID, recipientDID, artifactCID)` and `FetchEscrowSplitCommitment(fetcher, splitID)` are exported helpers that derive the deterministic SplitID (where applicable), query the log, and return the decoded commitment struct. Recipients and verifiers use these before calling `VerifyAndDecryptArtifact` or `Reconstruct`.

### 5. Wire-format locks.

- **KFrag plaintext:** 196 bytes. 99 active bytes (ID + RKShare + VK compressed + BK compressed) + 97 reserved bytes (zero-enforced; non-zero reserved bytes reject at deserialize via `ErrKFragReservedBytesNonZero`).

- **CFrag wire:** 196 bytes. Carries `(E', VK, BK, ID, ProofE, ProofZ, Reserved)`. DLEQ commitments `R`, `R'` are not transmitted; the verifier reconstructs them from `(z, e)` via `R = zÂ·G - eÂ·VK` and `R' = zÂ·E - eÂ·E'`. Reserved bytes are zero-enforced identically to KFrag via the shared `assertReservedZoneZero` helper. The 33-byte delta from v7.5's 163-byte CFrag is the version discriminator: any 163-byte input to a v7.75 deserializer rejects at length check, and any 196-byte input to a v7.5 deserializer rejects at length check.

- **PREGrantCommitment wire:** `SplitID || M || N || M Ă— 33-byte compressed points`. Total size is `34 + 33*M` bytes, maximum 8,449 bytes at M=255.

- **EscrowSplitCommitment wire:** `SplitID || M || N || dealerDID length-prefixed || M Ă— 33-byte compressed points`. Matches the PRE format with the dealer identity field included.

- **Share V2:** 131 bytes. Identical wire length to V1; V1 shares populate BlindingFactor and CommitmentHash with zeros, V2 shares populate them with polynomial evaluations.

Any deviation in wire length rejects at deserialize. v7.5 and v7.75 wire formats are not cross-readable by design, and migration between them is not supported: v7.5 grants complete their lifecycle on their original CFrag format, and new grants post-upgrade use v7.75 format exclusively.

### 6. Mutation-audit discipline is the permanent regression gate.

Every `muEnable*` constant in the SDK has:

- A colocated `*.mutation-audit.yaml` registry in the same directory as the source file, declaring the constant, its description, and the binding tests that genuinely fail when the constant is flipped.
- A binding test that flipping the constant to `false` causes a specific test assertion to fail (not merely "some test somewhere fails" â€” the registry names the specific tests).
- A dated entry in `docs/audit/mutation-audit-log.md` recording the last successful mutation run.

The runner at `cmd/audit-v775` enforces the discipline automatically. For each gate in each registry: flip the constant via `go/ast` rewrite, run the listed tests with `go test -run '^TestName$'`, assert non-zero exit, restore the constant, run the tests again, assert zero exit. The audit log is append-only and committed to the repo. `make audit-v775` runs the full sweep.

The CI guard catches three failure modes:

1. A gate flipped off in a diff (`grep -rn 'muEnable.*=\s*false' --include='*.go'` returns non-zero).
2. A gate deleted from source without its registry entry being removed (the validator detects a declared constant that no longer exists in the named file).
3. A registry entry pointing at a deleted constant, a deleted test, or a renamed test (the validator detects drift in either direction).

Target files span cryptographic primitives (Tier 1 and Tier 2) and composition layer (Tier 3) â€” roughly thirteen files at Phase C close. Tier 1 covers `crypto/artifact/pre.go`, `crypto/artifact/pre_grant_commitment.go`, `crypto/escrow/vss_v2.go`, `crypto/escrow/split_commitment.go`, and the M4 transcript DST. Tier 2 covers `core/vss/pedersen.go`, `core/vss/h_generator.go`, `crypto/escrow/verify_share.go`, `crypto/signatures/entry_verify.go`, `crypto/signatures/bls_verifier.go`, `core/envelope/serialize.go`, `core/smt/verify.go`. Tier 3 covers `verifier/cosignature.go`, `verifier/fraud_proofs.go`, `verifier/cross_log.go`, `verifier/authority_evaluator.go`, `witness/verify.go`, `witness/equivocation.go`, `lifecycle/artifact_access.go`, `lifecycle/recovery.go`.

### 7. Phase C source corrections.

Two non-crypto source changes land in Phase C to close defects surfaced during the audit read-pass.

- **`ConstraintState` enum restructure.** The enum in `verifier/authority_evaluator.go` is restructured to place `ConstraintUnclassified = iota` at the zero position, with `ConstraintActive`, `ConstraintPending`, and `ConstraintOverridden` shifted up by one. The classification loop's skip-guard becomes load-bearing: `if allEntries[i].State != ConstraintUnclassified { continue }` now correctly skips entries already classified by snapshot-specific logic, where previously the guard `State != 0` was silently dead because `ConstraintActive` was itself the zero value. The restructure prevents a future refactor that "fixes" the apparent bug by making the guard work as advertised from silently reintroducing the constraint-laundering exploit. Callsite sweep across `verifier/`, `lifecycle/`, and `judicial-network/` migrates any numeric-literal comparisons to named constants; serialization code that encodes `ConstraintState` as a raw byte is reviewed for the shifted numeric values.

- **Snapshot evidence walk cap.** The `EvidencePointers` walk inside `EvaluateAuthority`'s snapshot shortcut is capped at `MaxSnapshotEvidencePointers = 256`. The envelope writer exempts snapshots from the admission-time `MaxEvidencePointers` cap (correctly, per the snapshot's role), which left the verifier walking whatever length a malicious authorized signer chose to publish. A 500,000-pointer snapshot would cause downstream light clients and verifier nodes to OOM during the fetch-and-deserialize loop. The cap closes the DoS vector while keeping the shortcut's O(A) optimization intact.

## Scope boundaries

v7.75 does not modify `builder/algorithm.go` or any Path A/B/C classifier logic. Earlier drafts proposed three "builder authorization fixes" sourced from a secondary security review; re-reading current source against that review refuted all three claims. `PriorAuthority` is the scope-observation pointer by protocol design (not a multiplex with target OCC). `processPathA` and `processPathB` already reject scope-entity targets. The new-leaf branch already validates scope membership. No `ScopeObservation` field is introduced. The ADR-001 four-part legitimacy test remains the gate for any future ControlHeader field proposal.

v7.75 does not change the ADR-004 scope-resolution primitive. `AuthorizedSetAtPosition` remains the single entry point for scope-at-position queries; Decision 52's error discipline and locality enforcement are unchanged. The Phase C snapshot hardening adds defense-in-depth to consumers of this primitive but does not alter the primitive itself.

v7.75 does not introduce new consumer trust inputs. The ADR-003 "caller-provides, SDK-validates" boundary holds; no new caller-supplied security-relevant inputs are added to `EvaluateConditions`, `EvaluateAuthority`, or the lifecycle layer. The lookup primitives `FetchPREGrantCommitment` and `FetchEscrowSplitCommitment` consume the log fetcher as input but derive the SplitID from public grant context or from the share's own SplitID field; they do not accept a caller-supplied SplitID as a trust input.

v7.75 remains at protocol version 7. The commitment schemas are schema-layer additions; existing v7 commentary-entry machinery covers their admission and witness flow without protocol-level changes.

v7.75 is a complete SDK release. There is no Phase D. Operator deployments consume the commitment-entry surface (structs, builders, schemas, atomic emission, lookup primitives) as shipped, without needing SDK extensions. External cryptographic audit is scheduled as a separate work item after Phase C closes but is not a phase of v7.75 itself; its outcomes drive subsequent ADRs if they surface issues.

## Consequences

### Positive

Substitution attacks on escrow shares and PRE KFrags are cryptographically detected at distribution and reconstruction time rather than surfacing as silent application-layer failures. A malicious escrow node substituting its share, or a coalition of proxies jointly substituting their KFrags, is caught at the Pedersen polynomial-consistency check before Lagrange combination recovers the wrong secret.

DST discipline is uniform across the SDK. Boundary-shift attacks across identifier constructions are structurally excluded: no construction that hashes variable-length fields can produce colliding outputs for distinct input tuples under the universal length-prefix rule.

The mutation-audit registries make security gates machine-verifiable at the scale the composition layer demands. Beyond the first dozen `muEnable*` constants, hand-audit becomes untrustworthy and dated log entries drift; the runner eliminates both failure modes.

The classification loop in `EvaluateAuthority` has a load-bearing guard that resists accidental regression. A future developer reading the guard and "fixing" it can no longer silently reintroduce the constraint-laundering exploit.

The commitment-entry surface is complete within the SDK. Operator deployments get the full lifecycle without having to extend the SDK or wait for a downstream phase. Deterministic PRE SplitIDs mean recipients, proxies, and auditors can locate commitment entries from public grant context alone.

Forking scenarios benefit: a future auditor verifying a historical credential uses the same commitment-entry chain the originating infrastructure used, making cross-jurisdiction verification cryptographic rather than infrastructural.

### Negative

CFrag wire grows from 163 to 196 bytes (+20%). Each escrow split and PRE grant emits one additional Path A entry, incurring one additional witness-cosignature coverage cost per split or grant. Verification adds one Pedersen check per share and per CFrag; on secp256k1 this is in the sub-millisecond range per check.

PRE re-granting to the same recipient for the same artifact requires artifact key rotation. Prior workflows that assumed free re-grants must adopt the revoke-rotate-re-grant pattern. This is by design: silent re-grants were never observable on-log, and the deterministic SplitID forces them to be.

The `ConstraintState` enum shift requires a callsite sweep. Any code that compared against numeric literals rather than named constants needs to migrate; any serialization of `ConstraintState` as a raw byte needs a version tag or migration.

### Neutral

V1 shares and v7.5 CFrags remain readable and reconstructible on their original paths. v7.75 is additive at the API level; domain applications benefit without modification. No protocol version bump; the commitment schemas ride on the existing v7 commentary-entry machinery.
