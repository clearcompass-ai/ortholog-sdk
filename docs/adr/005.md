## Draft 2 Changes from Draft 1

Three holes identified in internal peer review are fixed in this draft:

1. **§4 (Generator Derivation):** specification now matches the try-and-increment construction actually implemented in `core/vss/h_generator.go`. Draft 1 specified RFC 9380 SSWU; the implementation rejected SSWU for auditor-burden reasons. ADR is now authoritative.
2. **§5.2 (Fiat-Shamir Transcript):** replaced `H(msg), U, U'` (Schnorr-style Umbral variables that do not exist in the Umbral PRE CFrag construction) with `E, E'` (capsule ephemeral and re-encrypted capsule ephemeral, matching `crypto/artifact/pre.go`).
3. **§6.2 (PRE Grant SplitID):** removed `nonce` and `timestamp` from SplitID input. The SplitID is now fully deterministic from the (grantor, recipient, artifact) tuple, making the commitment entry discoverable by recipients and proxies who do not know the dealer's internal randomness.
4. **§7.4 (Uniqueness enforcement):** replaced the builder-side first-write-wins logic (which is structurally impossible for commentary entries that do not produce SMT leaves) with verifier-side equivocation-detection.
5. **§13.1 (Cost analysis):** corrected off-by-one in scalar-multiplication count. For the RHS computation `∏ C_j^{i^j}` where `j ∈ {0, ..., M-1}`, only `j ≥ 1` requires a scalar multiplication (since `i^0 = 1`); total is `M + 1` scalar muls, not `M + 2`.

## 0. Executive Summary

The Ortholog SDK as of v7.5 contains two subtle cryptographic substitution vectors in its threshold-secret subsystems. In the escrow subsystem, an adversarial share holder can substitute a well-formed but incorrect share value and produce a silent reconstruction failure that surfaces only when the reconstructed secret is used — potentially decades after the substitution. In the Umbral proxy re-encryption (PRE) subsystem, a coalition of compromised proxies can produce CFrags that pass the standard Discrete Log Equivalence (DLEQ) proof without corresponding to any legitimate KFrag from the actual delegation split.

Both vectors follow from the absence of a cryptographic binding between individual threshold shares and their originating polynomial. Neither vector is prevented by DLEQ, threshold checks, or share-format validation. Both are exploitable in practice with modest effort by parties that already hold one legitimate share or KFrag.

This ADR specifies the cryptographic construction that closes both vectors in v7.75:

1. **Pedersen Verifiable Secret Sharing (VSS)** replaces plain Shamir for both subsystems. Each threshold split publishes a commitment to the secret polynomial and a commitment to an independent blinding polynomial. The commitments are published as on-log commentary entries under two new schemas (`escrow-split-commitment-v1`, `pre-grant-commitment-v1`). Share verification — at distribution time and at reconstruction time — reduces to a single polynomial-consistency check per share, costing one group-exponentiation per commitment-degree.

2. **Nothing-up-my-sleeve generator derivation** via try-and-increment from a locked seed string produces the Pedersen second generator `H` deterministically and verifiably. No trusted setup ceremony. No hidden discrete log relation between `G` and `H`.

3. **CFrag Pedersen binding** extends the Umbral PRE wire format with a 33-byte field `BK_i` per CFrag. Verification now requires both the existing DLEQ check and a Pedersen consistency check. Forged CFrags fail the Pedersen check even when the forger can construct a valid DLEQ proof.

4. **Independent blinding polynomials per VSS event.** Every escrow split and every PRE grant generates its own polynomial pair. No blinding material is shared between events, between subsystems, or across time. This is structurally enforced by the `core/vss/pedersen` package API.

5. **Fiat-Shamir transcript hardening.** The DLEQ challenge hash absorbs the commitment set and the Pedersen commitment `BK_i` before the standard DLEQ inputs. This prevents adaptive selection of BK_i to manipulate challenge values.

The construction assumes an honest dealer with respect to polynomial consistency. This is an appropriate assumption for v7.75's deployment model, where dealers are SDK-controlled processes (provision, key rotation, grant emission). Client-generated key flows require additional machinery (NIZK proofs of polynomial correctness) and are deferred to a future release.

Security depends on the hardness of the discrete logarithm problem on secp256k1 (classical adversary), SHA-256 collision resistance, and the standard security assumptions of AES-256-GCM. Post-quantum security is not claimed.

## 1. Context and Problem Statement

### 1.1. The Ortholog SDK's Threshold-Secret Subsystems

The Ortholog SDK operates two subsystems in which a secret is threshold-shared:

**The escrow subsystem** (`crypto/escrow/`, `exchange/identity/mapping_escrow.go`, `lifecycle/recovery.go`). Secrets relevant to log recovery — including mapping-record master keys and delegation-key unwrap material — are split via Shamir secret sharing over GF(2^256) and distributed across a set of escrow nodes. Reconstruction requires `M`-of-`N` shares. Use cases include recovery after key compromise, jurisdiction transfers (scenario 3 in the forking document), and scheduled delegation-key rotation.

**The Umbral PRE subsystem** (`crypto/artifact/pre.go`, `lifecycle/artifact_access.go`). When a data owner grants a recipient decryption access to an artifact, the owner's artifact key is re-encryption-shared across a set of proxies via Umbral proxy re-encryption. Each proxy holds a KFrag. When the recipient requests access, each proxy produces a CFrag; `M`-of-`N` CFrags combine via Lagrange interpolation to yield a re-encryption key the recipient can use.

Both subsystems use `M`-of-`N` threshold secret sharing as their core primitive. Both are security-critical: compromise of the escrow subsystem leaks mapping records across DID namespaces; compromise of the PRE subsystem leaks artifact contents to unauthorized recipients.

### 1.2. The Substitution Vectors

The v7.5 constructions of both subsystems have a cryptographic gap: **shares and KFrags are not cryptographically bound to their originating polynomial.** The share-format layer validates the SplitID and field-tag but performs no consistency check against the polynomial from which the share was derived.

**Vector 1 — Escrow share substitution.** An escrow node holding a legitimate share `(i, s_i)` for a split with SplitID `SID` can substitute the share value: replace `s_i` with any field element `s'_i ≠ s_i`. The substituted share `(i, s'_i)` still has the correct SplitID, passes `ValidateShareFormat`, and enters Lagrange interpolation. The resulting reconstructed secret is incorrect, but the reconstruction itself does not signal an error.

Detection of the substitution happens at use time. If the reconstructed secret is an AES key for a sealed artifact, the AES-GCM authentication check fails at decryption. If the reconstructed secret is a delegation-key-unwrap secret, the unwrap produces gibberish. In both cases, the detection occurs after the reconstruction, at an application layer removed from the cryptographic subsystem — and critically, with no information about which escrow node contributed the bad share.

For sealed records stored for decades, the substitution attack is genuinely silent. A malicious escrow node can substitute a share during distribution, and the fact that the secret cannot be reconstructed is not discoverable until a recovery is attempted. By that time, the evidentiary trail for identifying the malicious node may be gone.

**Vector 2 — Umbral KFrag substitution.** A proxy holding a legitimate KFrag can substitute its KFrag with one produced from a different delegation scalar. The substituted KFrag produces a CFrag for which the proxy can construct a valid DLEQ proof. The DLEQ proof shows that the proxy used a consistent key pair — but it does not show that the key pair corresponds to the actual delegation split.

A coalition of `M` compromised proxies can substitute their KFrags jointly to any scalar they agree on, produce `M` mutually consistent CFrags with valid DLEQ proofs, combine these CFrags via Lagrange to recover an attacker-chosen re-encryption key, and decrypt any artifact for which they held CFrags. The legitimate owner and recipient see no signal of the attack: DLEQ verification passes, threshold is satisfied, decryption succeeds with the attacker's chosen key.

This is a genuine attack, not a theoretical concern. The standard Umbral PRE construction (Nuñez et al., 2018) assumes proxies are honest-but-curious and does not defend against a coalition of compromised proxies coordinating KFrag substitution. Ortholog's deployment model explicitly contemplates proxy sets that may contain adversarial members — hence the K-of-N threshold in the first place.

### 1.3. Why Existing Defenses Are Insufficient

The v7.5 codebase has several defenses that *feel* like they should address these vectors but do not:

- **SplitID binding.** Ties shares to a specific split but does not constrain the share value within that split.
- **Field tag (`0x01` = GF(2^256)).** Prevents cross-field reconstruction (pilot Exp 7) but does not detect same-field substitution.
- **DLEQ on CFrags.** Proves internal consistency of the proxy's own key pair but not correspondence to the delegation split.
- **AES-GCM authentication tag.** Detects incorrect reconstruction *at decryption time*, not at reconstruction time. Surfaces as an application-layer error with no attribution.
- **K-of-N threshold.** Tolerates passive failures of up to `N - K` nodes. Does not defend against active substitution by any number of contributing nodes.

None of these mechanisms provide what is actually needed: a cryptographic commitment at split time to the specific polynomial from which shares will be generated, and a mechanism for any party to verify a share is consistent with that commitment.

### 1.4. The Construction Space

Threshold secret sharing admits multiple forms of verifiability:

**Feldman VSS** (Feldman 1987) publishes commitments `C_j = G^{a_j}` to the secret polynomial's coefficients. Share verification is `G^{s_i} = ∏_j C_j^{i^j}`. This works but reveals `G^{secret}` publicly. In contexts where `G^{secret}` is sensitive (e.g., the secret is a private scalar whose public counterpart must not be discoverable), Feldman's information leakage is unacceptable.

**Pedersen VSS** (Pedersen 1991) publishes commitments `C_j = G^{a_j} · H^{b_j}` where `a_j` are the coefficients of the secret polynomial and `b_j` are coefficients of an independent random blinding polynomial. Share verification is `G^{a(i)} · H^{b(i)} = ∏_j C_j^{i^j}`. The construction is information-theoretically hiding with respect to the secret: the commitments reveal *nothing* about `G^{secret}` beyond what is implied by computational assumptions on `G` and `H`.

**NIZK-backed VSS** (various) adds non-interactive zero-knowledge proofs of polynomial consistency, defending against adversarial dealers. Cost: proof sizes of several kilobytes per share and verification times in the tens of milliseconds.

For the escrow subsystem, Feldman's `G^{secret}` leakage is unacceptable because the reconstructed secret is itself a cryptographic key (AES or delegation-unwrap). Pedersen's hiding property preserves the secrecy of `G^{secret}`.

For the PRE subsystem, Feldman would reveal `G^{rk}` where `rk` is the re-encryption key derived from `sk_owner` and `pk_recipient`. This is sensitive in Umbral's construction: `G^{rk}` combined with other public material enables computation of intermediate values in the re-encryption. Pedersen's hiding preserves security.

NIZK-backed VSS is strictly stronger than Pedersen (defends against adversarial dealers) but significantly more expensive. For v7.75, the honest-dealer assumption is appropriate (see §11) and NIZK-backed constructions are deferred.

**Conclusion: Pedersen VSS is the correct construction for v7.75 in both subsystems.**

## 2. Design Goals and Non-Goals

### 2.1. Goals

1. **Share substitution is cryptographically detected.** After v7.75, a substituted share or KFrag fails verification at distribution time (by escrow node / recipient) and at reconstruction time (by requester / delegatee). No silent failure reaches application-layer detection.

2. **Additive deployment.** v7.75 is a non-breaking change for existing v7.5 consumers. V1 shares at rest remain readable. V1 reconstruction paths remain functional. Domains do not need to modify application code to benefit from v7.75.

3. **Verifiable by any party.** Share verification requires only publicly available information: the commitment set (from the on-log commentary entry) and the share itself. No interactive protocol. No trusted party.

4. **Information-theoretic hiding of `G^{secret}`.** The commitment set reveals nothing about `G^{secret}` computable by any adversary bounded by the discrete log assumption on secp256k1.

5. **Nothing-up-my-sleeve generator derivation.** The Pedersen generator `H` is derived deterministically from a locked seed string. No trusted setup ceremony.

6. **Independent blinding per VSS event.** Every split and every grant uses its own polynomial pair. No reuse. Structurally enforced by API.

7. **Domain separation.** Escrow splits and PRE grants use distinct schemas, distinct SplitID DSTs, and distinct generator contexts. No cross-subsystem commitment reuse.

8. **Wire-format stability.** Once v7.75 ships, the commitment entry schemas and CFrag wire format are locked. Future changes require a new protocol version (v7.76 or v8), not retrofitting v7.75.

9. **Reproducibility by independent implementations.** The specification is exhaustive enough that a future Rust or TypeScript implementation can produce byte-identical commitment entries, CFrags, and challenge hashes.

### 2.2. Non-Goals

1. **Post-quantum security.** The construction is secure under the discrete log assumption on secp256k1, which is classical. Post-quantum adversaries are out of scope. Migration to post-quantum primitives is a protocol-level effort for a future release.

2. **Defense against adversarial dealers.** The honest dealer assumption is load-bearing. A dealer producing inconsistent commitments (commitments not matching the polynomial) is detectable at first-honest-party verification time, but not prevented. Full dealer-malice defense requires NIZKs and is deferred.

3. **Client-generated keys.** When the client is the dealer (e.g., a user-controlled wallet splitting its own secret), the honest-dealer assumption does not hold by default. v7.75 does not provide client-side dealer malice protection. Scope for v7.76 or later.

4. **Optimized batch verification.** v7.75 uses naive O(M) verification per share. For large M (≥ 10), multi-scalar multiplication (MSM) or pre-computation is more efficient. Optimization is deferred to v8 if operational scale demands it.

5. **New subsystem integration.** This ADR specifies Pedersen binding for the existing escrow and PRE subsystems. New subsystems (e.g., threshold signatures in v8) may reuse the `core/vss/pedersen` primitive, but the specific integration is out of scope for v7.75.

6. **Migration of v1 shares.** Shares produced under v7.5's V1 format remain readable forever. They are NOT upgraded to V2 retroactively. Callers wanting V2 properties must create a new split; this is expected and acceptable since V2 is additive.

## 3. The Construction

### 3.1. Notation and Group Setup

Let `E = secp256k1` denote the standard secp256k1 elliptic curve with generator `G` of prime order `n` (approximately `2^256 - 2^32 - 977`). Let `F_n = Z / nZ` denote the scalar field.

Let `H ∈ E` be the Pedersen second generator, derived deterministically by the construction in §4. The discrete log of `H` with respect to `G` is unknown (by the nothing-up-my-sleeve argument) and assumed hard (by the discrete log assumption on secp256k1).

A Pedersen commitment to a pair `(a, b) ∈ F_n × F_n` is the point `Com(a, b) = a·G + b·H ∈ E`.

A threshold split of a secret `s ∈ F_n` with threshold `M` produces:

- A secret polynomial `f(x) = a_0 + a_1 x + ... + a_{M-1} x^{M-1}` where `a_0 = s` and `a_1, ..., a_{M-1}` are sampled uniformly from `F_n`.
- A blinding polynomial `r(x) = b_0 + b_1 x + ... + b_{M-1} x^{M-1}` where `b_0, ..., b_{M-1}` are sampled uniformly from `F_n`, **independent** of `f`.
- A commitment set `C = (C_0, C_1, ..., C_{M-1})` where `C_j = Com(a_j, b_j) = a_j·G + b_j·H`.
- A set of `N` shares `(i, f(i), r(i))` for `i ∈ {1, 2, ..., N}`.

By linearity of the commitment map, the following identity holds for every valid share:

```
f(i)·G + r(i)·H  =  ∑_{j=0}^{M-1} (i^j mod n) · C_j
```

(Note: the literature typically writes Pedersen commitments multiplicatively as `G^a · H^b`. This ADR uses additive notation throughout since the group is an elliptic curve; implementers should read `·` as scalar multiplication of a point and `+` as point addition.)

### 3.2. Split Algorithm (Escrow Subsystem)

**Input:** secret `s ∈ F_n` (derived from the 32-byte secret via big-endian integer interpretation, reduced mod `n`), threshold `M`, total `N`, dealer DID (NFC-normalized, UTF-8 encoded), random 32-byte nonce.

**Algorithm:**

```
1.  Sample a_1, ..., a_{M-1} ← F_n \ {0} uniformly at random.
2.  Sample b_0, b_1, ..., b_{M-1} ← F_n \ {0} uniformly at random.
3.  Compute f(x) = s + a_1·x + ... + a_{M-1}·x^{M-1}.
4.  Compute r(x) = b_0 + b_1·x + ... + b_{M-1}·x^{M-1}.
5.  Compute C_j = a_j·G + b_j·H for j ∈ {0, ..., M-1}, with a_0 = s.
6.  Compute SplitID = SHA-256(
        "ORTHOLOG-V7.75-ESCROW-SPLIT"  ||
        len(dealerDID, 2 bytes BE)     ||
        dealerDID                      ||
        nonce
    )
7.  For i ∈ {1, ..., N}:
        Compute f_i = f(i) mod n.
        Compute r_i = r(i) mod n.
        Compute CommitmentHash = SHA-256(
            BE_uint32(M) || C_0 || C_1 || ... || C_{M-1}
        )
        Emit share (Version=V2, SplitID, Index=i, Value=f_i,
                    BlindingFactor=r_i, CommitmentHash)
8.  Emit commitment-entry payload:
        (SplitID, M, N, dealerDID, C_0, C_1, ..., C_{M-1}).
9.  Return shares, commitment set C, SplitID.
```

The commitment entry is published as a Path A commentary entry under schema `escrow-split-commitment-v1` before the first share is distributed to any escrow node. This is the publication invariant (see §7).

Zero scalars are rejected at coefficient sampling: a zero top-degree coefficient would reduce the effective polynomial degree and enable reconstruction from `M-1` shares, a quorum-reduction attack. Rejecting zero at every coefficient position is strictly conservative.

### 3.3. Share Verification

**Input:** share `(Version=V2, SplitID, i, f_i, r_i, CommitmentHash)`, commitment set `C`.

**Algorithm:**

```
1.  Verify Version == V2.
2.  Verify BlindingFactor r_i is nonzero (rejects unpopulated V2 shares).
3.  Recompute expected_hash = SHA-256(
        BE_uint32(M) || C_0 || C_1 || ... || C_{M-1}
    )
    Verify CommitmentHash == expected_hash.
    (Binds the share to a specific commitment set; rejects cross-split mixing.)
4.  Reduce f_i and r_i mod n. (Canonical representation for curve ops.)
5.  Compute LHS = f_i·G + r_i·H.
6.  Compute RHS = ∑_{j=0}^{M-1} (i^j mod n) · C_j.
    For j=0, the factor is C_0 (since i^0 = 1; no scalar mul).
    For j ≥ 1, the factor is (i^j mod n) · C_j.
    Accumulate in affine or Jacobian coordinates.
7.  Verify LHS == RHS (as points on secp256k1).
8.  On any inequality or invalid curve point: reject.
```

If the check passes, the share is cryptographically certified as consistent with the published commitment set. If the check fails, one of the following has happened:

- Share value or blinding factor was substituted after distribution (attack detected).
- Commitment hash does not match the commitment set being verified against (wrong split or corrupted commitment entry).
- Dealer published an inconsistent commitment set relative to the actual polynomials (dealer malice, detected at first-honest-verifier).

### 3.4. Reconstruction

**Input:** set of `M` or more verified shares, commitment set `C`.

**Algorithm:**

```
1.  For each share in the input set, perform share verification (§3.3).
    Reject if any share fails. Reject if duplicate Index.
2.  Select the first M shares from the verified set.
3.  Compute secret = ∑_i λ_i · f_i mod n
    where λ_i = ∏_{j≠i} (j / (j - i)) mod n are the standard Lagrange
    coefficients evaluated at x=0.
4.  Convert secret (a scalar in F_n) back to the 32-byte representation
    (big-endian encoding, left-padded to 32 bytes).
5.  Return the reconstructed secret.
```

The pre-verification step (1) is **mandatory**. Reconstruction from unverified shares recovers the wrong secret under substitution attack — and recovers it without error indication. The pre-verification is what transforms silent failure into loud rejection.

    bkX, bkY := curve.ScalarMult(H_x, H_y, padScalar(bi))
    kfrags[i].BK = compressedPoint(bkX, bkY)
    // Zeroize bi before the next iteration.
    bi.SetInt64(0)
    zero32(&s.BlindingFactor)
}The zero32(&s.BlindingFactor) call zeros the 32-byte array in the vss.Share, preventing subsequent inspection of the share list from recovering blinding factors. bi.SetInt64(0) is best-effort on the *big.Int per the zeroization-limits note in the package-level godoc.

## 4. Generator Derivation

### 4.1. The Problem

Pedersen commitments require a second generator `H` such that `log_G(H)` is unknown. If the dealer knows `log_G(H) = x`, the commitment `C = a·G + b·H = (a + xb)·G` is equivalent to a Feldman commitment with coefficient `a + xb` — the hiding property is lost. Worse, the dealer can equivocate: given any commitment `C`, they can produce a different `(a', b')` opening with `a' - a = x(b - b')`.

Historical deployments of Pedersen-based schemes have addressed this in two ways:

**Trusted setup ceremony.** Multiple parties contribute to the generation of `H`, with the property that `log_G(H)` is unknown as long as at least one contributor was honest. Used by Zcash (Powers of Tau). Operationally heavyweight; requires coordination and trust.

**Nothing-up-my-sleeve derivation.** The generator `H` is derived deterministically from a public, verifiable input (a hash of a well-known string, a mathematical constant, etc.). Anyone can independently reproduce the derivation and verify `H`. The discrete log of `H` is hard *assuming the hash function is sufficiently random* and that no one has a way to invert the derivation.

For v7.75, trusted setup is rejected: the SDK has no ceremonial infrastructure, and a trusted setup ceremony would be the largest operational burden in any ClearCompass deployment. Nothing-up-my-sleeve derivation is strictly preferred.

### 4.2. The Locked Construction

The Pedersen generator `H` for v7.75 is derived via **try-and-increment** from a locked seed string:

```
seed = "ortholog/core/vss/pedersen/h-generator/v1"

counter ← 0
loop forever:
    digest = SHA-256(seed || BE_uint32(counter))
    x = digest mod p          (where p is the secp256k1 field prime)
    if y exists with (x, y) on secp256k1:
        y ← the numerically smaller of the two roots  (canonical tie-break)
        return H = (x, y)
    counter ← counter + 1
```

**p** (secp256k1 field prime) = `2^256 - 2^32 - 977`.

The derivation terminates with probability ≈ 1/2 on each iteration (roughly half of `F_p` elements are x-coordinates of curve points). In expectation, termination occurs at counter ≤ 2. A bounded loop (maximum 256 iterations) is implemented for paranoia; the bound has never been approached.

### 4.3. The Locked Seed

```
seed = "ortholog/core/vss/pedersen/h-generator/v1"
```

As ASCII bytes (41 bytes):

```
0x6F 0x72 0x74 0x68 0x6F 0x6C 0x6F 0x67  (ortholog)
0x2F                                     (/)
0x63 0x6F 0x72 0x65                      (core)
0x2F                                     (/)
0x76 0x73 0x73                           (vss)
0x2F                                     (/)
0x70 0x65 0x64 0x65 0x72 0x73 0x65 0x6E  (pedersen)
0x2F                                     (/)
0x68 0x2D 0x67 0x65 0x6E 0x65 0x72 0x61 0x74 0x6F 0x72  (h-generator)
0x2F                                     (/)
0x76 0x31                                (v1)
```

Total length: 41 bytes.

The suffix `v1` is specifically reserved for the generator derivation version. If a future revision changes the derivation (different seed string, different hash-to-curve procedure, different curve), the seed suffix increments to `v2`. The seed lifecycle discipline is specified in §14.

### 4.4. Why Try-and-Increment Instead of RFC 9380 Hash-to-Curve

RFC 9380's SSWU map produces a constant-time, statistically-unbiased map from arbitrary inputs to curve points. This is valuable in per-message hashing protocols (BLS signatures, VOPRFs, hash-based commitments to user input) where:
- Timing leakage about a secret input must be prevented.
- Statistical bias could be exploited across many hash invocations.

Neither property is relevant for a one-time startup derivation of a protocol constant:
- The derivation runs once per process lifetime and is independent of any secret data. There is no timing-side-channel surface.
- Statistical bias on a single output point is not meaningful. The requirement is that `log_G(H)` is hard, which requires only that `H` is some unstructured point on the curve, not that it is drawn from a perfectly uniform distribution.

Try-and-increment is the standard construction for this purpose. Specifically:
- **Bitcoin Core** uses try-and-increment to derive `secp256k1_generator_h` for Confidential Transactions range proofs.
- **Zcash Sapling** uses try-and-increment to derive Pedersen commitment generators for the SaplingValue commitment scheme.
- **Monero** uses try-and-increment to derive `H = hash_to_point(G)` for its Pedersen commitments.

Adopting try-and-increment:
- Avoids ~200 lines of SSWU map implementation the auditor would need to verify.
- Avoids an external library dependency (the SDK's current cryptographic libraries — `decred/secp256k1/v4`, `consensys/gnark-crypto/bls12-381`, Go stdlib — do not expose secp256k1 SSWU).
- Matches established precedent in the closest-analogous deployments.

The decision was made after ADR-005 Draft 1 (which specified RFC 9380 SSWU). Draft 2 aligns the ADR with the implementation.

### 4.5. Independence of `H` from `G`

**Claim:** Under the random oracle model for SHA-256, `log_G(H)` is uniformly distributed in `F_n` and uncomputable by any polynomial-time adversary, including the author of the seed constant.

**Argument.** The argument has four steps:

1. **Seed independence from G.** The seed `"ortholog/core/vss/pedersen/h-generator/v1"` is a human-readable text string chosen before `G`'s coordinates were consulted. A malicious author who wanted `H = k·G` for some known `k` would need to choose a seed that, when hashed with a specific counter, produces `x = (k·G).x`. That reduces to second-preimage resistance on SHA-256, which is assumed infeasible.

2. **Derivation independence from G.** The try-and-increment procedure uses only the seed and the counter as inputs. The derivation does not reference `G`'s coordinates. `H` is therefore computed by a process that has no knowledge of `G`'s discrete structure.

3. **Uniform distribution.** Under the random oracle assumption for SHA-256, each candidate x-coordinate is a uniformly random element of `F_p`. Approximately half of those elements correspond to on-curve x-coordinates (the other half are quadratic non-residues). The first surviving candidate x, when combined with its canonical y (smaller of the two roots), produces a uniformly random point on the curve. Since secp256k1 has prime order, this is equivalent to `H = k·G` for a uniformly random `k ∈ F_n`.

4. **ECDLP hardness.** Computing `k` from `H` is the secp256k1 discrete logarithm problem for a uniformly random point. Under the standard ECDLP assumption on secp256k1, this is infeasible for polynomial-time adversaries at the 128-bit security level.

**Review discipline.** Step 1 is the load-bearing step. The seed must be chosen BEFORE the derivation is run. Auditors can verify this from git history: the seed constant was committed to the repository at the same time as the derivation code, before any test vector was computed. Any retroactive modification of the seed to produce a desired `H` would require rewriting history in a way that leaves forensic traces.

### 4.6. Reproducibility

Any implementation can reproduce `H` deterministically:

1. Construct the 41-byte seed string: `"ortholog/core/vss/pedersen/h-generator/v1"`.
2. For counter = 0, 1, 2, ...:
   a. Compute `digest = SHA-256(seed || BE_uint32(counter))` (45 bytes of input).
   b. Compute `x = digest mod p`.
   c. Test whether `y^2 = x^3 + 7 mod p` has a solution. If yes: select `y` as the smaller of the two roots, return `(x, y)`. If no: increment counter.

Test vectors in §17 pin the exact expected byte sequences: the counter value at which termination occurs, the `x` and `y` coordinates of `H`, and a SHA-256 digest of the encoded point for quick equality-checking.

## 5. Fiat-Shamir Transcript Specification

### 5.1. The Problem

The Umbral PRE CFrag DLEQ proof is a non-interactive zero-knowledge argument for the statement:

> There exists a scalar `rk_i` such that `VK_i = rk_i · G` AND `E' = rk_i · E`.

Here `G` is the secp256k1 generator, `E` is the capsule's ephemeral point, and `VK_i`, `E'` are the respective outputs of scalar multiplication by `rk_i`. The proof is constructed in the Fiat-Shamir paradigm: the challenge is derived as `e = Hash(transcript)` where the transcript contains the public parameters, the commitments, and the statement.

When v7.75 adds Pedersen binding via `BK_i`, the DLEQ transcript must absorb both the Pedersen commitment set `C` and the `BK_i` value. Otherwise, an adaptive adversary can choose `BK_i` post-hoc after seeing the challenge, breaking soundness.

**Attack without transcript absorption.** Suppose the DLEQ transcript covers only `(G, E, VK_i, E', R, R')` (the Umbral public values and DLEQ commitments) but NOT `BK_i` or `C`. An adversary could:

1. Construct a DLEQ proof for some adversarial `rk'_i` (not from the legitimate split) and `VK'_i = rk'_i · G`.
2. After observing the DLEQ challenge `e = Hash(G, E, VK'_i, E'', R, R')`, compute what `BK'_i` must be such that the Pedersen check `VK'_i + BK'_i = ∑ i^j · C_j` passes. This is a single algebraic step: `BK'_i = ∑ i^j · C_j - VK'_i`.
3. Publish `(VK'_i, BK'_i)` together with the DLEQ proof.

Both checks pass individually; the combined system accepts an adversarial CFrag.

**Defense.** Absorbing `BK_i` and `C` into the transcript before computing `e` means that changing either value changes `e`, which invalidates any DLEQ proof constructed for a different value. The adversary cannot both (a) choose `BK_i` to make the Pedersen check pass AND (b) have a valid DLEQ proof — the DLEQ proof was bound to a specific `BK_i` at challenge-computation time.

### 5.2. The Locked Transcript Format

The DLEQ challenge `e` is computed as:

```
e = SHA-256(
    "ORTHOLOG-V7.75-DLEQ-CHALLENGE-v1" + 0x00 padding   (32 bytes; 29 ASCII + 3 null)
    || BE_uint32(len(commitment_set))                    (4 bytes)
    || C_0 (compressed, 33 bytes)                        (33 bytes; Pedersen commitment)
    || C_1 (compressed, 33 bytes)                        (33 bytes)
    || ...
    || C_{M-1} (compressed, 33 bytes)                    (33 bytes)
    || BK_i (compressed, 33 bytes)                       (33 bytes; blinding commitment)
    || VK_i (compressed, 33 bytes)                       (33 bytes; verifier key rk_i·G)
    || E (compressed, 33 bytes)                          (33 bytes; capsule ephemeral)
    || E_prime (compressed, 33 bytes)                    (33 bytes; re-encrypted E' = rk_i·E)
    || BE_uint64(index i)                                (8 bytes)
    || R (compressed, 33 bytes)                          (33 bytes; DLEQ commitment k·G)
    || R_prime (compressed, 33 bytes)                    (33 bytes; DLEQ commitment k·E)
)
```

The challenge `e` is interpreted as a scalar in `F_n` by reducing the 32-byte SHA-256 output mod n.

**Byte ordering is exhaustive and unambiguous.** Every field has a fixed length. Integers are big-endian. Curve points are 33-byte SEC 1 compressed encodings. No optional fields, no conditional encodings.

**DLEQ statement explicitly.** The proof establishes: knowledge of `rk_i ∈ F_n` such that `VK_i = rk_i · G` and `E' = rk_i · E`. The commitments `R = k · G` and `R' = k · E` for a fresh per-proof nonce `k` are the Fiat-Shamir commitments. The response is `z = k + e · rk_i mod n`, which the verifier checks via `z · G == R + e · VK_i` and `z · E == R' + e · E'`.

### 5.3. DST and Length Prefix

The 32-byte transcript prefix `"ORTHOLOG-V7.75-DLEQ-CHALLENGE-v1"` (29 ASCII characters, right-padded with `0x00 0x00 0x00` to 32 bytes) ensures that:
- The challenge hash is domain-separated from any other SHA-256 input in the SDK.
- No SHA-256 collision across protocols (BLS witness cosignatures, entry hashes, SplitIDs) can produce a valid challenge.

The `BE_uint32(len(commitment_set))` prefix serves two purposes:
- Disambiguates commitment sets of different sizes (M=3 vs M=5 cannot collide).
- Prevents length-extension attacks on the unbounded portion of the transcript.

### 5.4. Compressed Point Encoding

All points in the transcript are serialized using SEC 1 compressed encoding: 33 bytes of the form `0x02 || x` (for y even) or `0x03 || x` (for y odd). This is the standard compressed form used throughout the Ortholog codebase and is unambiguous: each point on secp256k1 has a unique 33-byte compressed representation.

### 5.5. Verification

To verify a DLEQ proof, the verifier computes the challenge `e` using the locked transcript format with:
- The publicly known commitment set `C` (from the on-log commitment entry, retrieved via the SplitID lookup per §6.2).
- The `BK_i` from the CFrag being verified.
- The `VK_i` from the CFrag.
- The capsule's `E` (from the re-encryption context).
- The `E'` from the CFrag.
- The index `i` from the CFrag.
- The commitments `R`, `R'` from the CFrag.

If the proof verifies with this `e`, the CFrag is cryptographically consistent (DLEQ check passes). The Pedersen check (§3.5) is performed separately; both must pass.

### 5.6. Non-Interference Proof Sketch

**Claim:** Absorbing `BK_i` and commitment set `C` into the DLEQ challenge preserves DLEQ's soundness.

**Proof sketch.** The DLEQ proof is a Schnorr-style non-interactive argument for the statement "there exists `rk_i` such that `VK_i = rk_i · G` and `E' = rk_i · E`." The standard Fiat-Shamir transform derives the challenge from a hash of the public parameters (`G`, `E`) and commitments (`R`, `R'`). Soundness holds in the random oracle model under ECDLP.

Absorbing additional public data into the hash does not weaken soundness. The hash function is still modeled as a random oracle; the output distribution is still uniform. Absorbing `BK_i` and `C` binds the challenge to specific values of `BK_i` and `C`, which prevents an adversary from choosing `BK_i` or `C` after observing the challenge.

No "malleability" is introduced: a valid proof for one `(BK_i, C)` pair is not a valid proof for a different `(BK'_i, C')` pair, because the challenge differs. This is precisely the property we want for the substitution defense.

A formal proof (simulation / extraction arguments) is omitted here as standard. See [Boneh-Shoup 2020, Chapter 19] for the full treatment of Fiat-Shamir soundness under random oracle composition.

**Claim:** Absorbing `BK_i` and `C` into the DLEQ challenge does not leak information about the blinding factor `b_i`.

**Proof sketch.** `BK_i = b_i · H` and `C = (a_j · G + b_j · H)_j`. These are commitments; computing `b_i` from them requires breaking Pedersen hiding, which reduces to the discrete log problem on secp256k1. The challenge hash is a public output of a deterministic function of public inputs; it carries no more information than the inputs themselves. Therefore, no information about `b_i` is leaked beyond what is already in `BK_i` and `C`.

## 6. Domain Separation and SplitID Construction

### 6.1. Escrow Split ID

```
SplitID_escrow = SHA-256(
    "ORTHOLOG-V7.75-ESCROW-SPLIT"  (27 bytes ASCII, UTF-8)
    || BE_uint16(len(dealerDID))   (2 bytes)
    || dealerDID (NFC-normalized, UTF-8)
    || nonce                       (32 bytes, uniformly random)
)
```

**DST bytes:**

```
0x4F 0x52 0x54 0x48 0x4F 0x4C 0x4F 0x47  (ORTHOLOG)
0x2D                                     (-)
0x56 0x37 0x2E 0x37 0x35                 (V7.75)
0x2D                                     (-)
0x45 0x53 0x43 0x52 0x4F 0x57            (ESCROW)
0x2D                                     (-)
0x53 0x50 0x4C 0x49 0x54                 (SPLIT)
```

The nonce allows legitimate re-splitting of the same master secret. During delegation-key rotation, for example, the same logical secret may be split multiple times with different escrow node sets; each split gets its own SplitID via the fresh nonce. The dealer records the nonce alongside the commitment entry for audit purposes.

### 6.2. PRE Grant ID (Deterministic)

```
SplitID_grant = SHA-256(
    "ORTHOLOG-V7.75-PRE-GRANT"       (24 bytes ASCII, UTF-8)
    || BE_uint16(len(grantorDID))    (2 bytes)
    || grantorDID (NFC-normalized, UTF-8)
    || BE_uint16(len(recipientDID))  (2 bytes)
    || recipientDID (NFC-normalized, UTF-8)
    || artifactCID                   (32 bytes; SHA-256 content address)
)
```

**DST bytes:**

```
0x4F 0x52 0x54 0x48 0x4F 0x4C 0x4F 0x47  (ORTHOLOG)
0x2D                                     (-)
0x56 0x37 0x2E 0x37 0x35                 (V7.75)
0x2D                                     (-)
0x50 0x52 0x45                           (PRE)
0x2D                                     (-)
0x47 0x52 0x41 0x4E 0x54                 (GRANT)
```

**The PRE grant SplitID is fully deterministic from the public grant context.** No nonce, no timestamp. Any party that knows `(grantorDID, recipientDID, artifactCID)` can reconstruct the SplitID and query the log for the commitment entry. This is essential for the protocol flow:

- **Recipient-initiated access.** The recipient knows the artifact they want to decrypt, their own DID, and the grantor's DID (from the grant notification or from knowledge of who owns the artifact). They derive the SplitID, fetch the commitment entry, and verify CFrags against it.
- **Proxy lookup.** Each proxy knows the artifactCID and the grant tuple. The proxy can fetch the commitment entry without a separate out-of-band discovery mechanism.
- **Third-party audit.** Any auditor with access to the grant context (from on-log records) can locate and verify the commitment entry.

### 6.2.1. Re-Grant Rotation Policy

Because SplitID_grant is deterministic, the dealer cannot produce two distinct commitment entries for the same `(grantor, recipient, artifact)` tuple. The second attempt is either (a) rejected at admission if the log operator enforces schema uniqueness, or (b) treated as equivocation by verifiers (§7.4).

This is an intentional constraint: genuine re-granting of access to the same artifact requires rotating one component of the tuple. The natural rotation is `artifactCID`:

1. Revoke the old grant by emitting a revocation entry on-log.
2. Re-encrypt the artifact with a fresh AES-256-GCM key via `ReEncryptWithGrant`. This produces a new artifactCID.
3. Issue a new grant against the new artifactCID. The SplitID is new; the commitment entry is new.

This forces revocation and re-grant to be observable on-log rather than silent. If a grantor needs to re-grant without rotating the artifact (for example, correcting an implementation bug in the previous grant), the lifecycle is: mark old grant revoked, rotate key via `ReEncryptWithGrant`, issue new grant. No grantor-level workflow supports "two grants for the same artifact to the same recipient simultaneously."

### 6.3. Why Distinct DSTs

Claim: `SplitID_escrow ≠ SplitID_grant` for any inputs producing the same SHA-256 output.

The DSTs differ in their first 27 vs 24 bytes. Any SHA-256 collision requires matching outputs from distinct inputs — and since the DSTs differ in their initial bytes, no pair of (escrow DST, inputs) and (grant DST, inputs) can produce identical input bytes. Therefore, collision-resistance of SHA-256 implies `SplitID_escrow ≠ SplitID_grant` for all input choices.

The practical consequence: a Pedersen commitment set published under `escrow-split-commitment-v1` cannot be presented as a valid commitment for a PRE grant, and vice versa. The schemas enforce this at the on-log layer; the SplitID enforces it at the cryptographic layer.

### 6.4. Nonce and Uniqueness (Escrow Only)

The 32-byte nonce in `SplitID_escrow` is sampled uniformly from `crypto/rand`. With probability `1 - 2^{-256}`, no two escrow SplitIDs collide across the entire deployment history of v7.75. Within a single escrow dealer's operation, the nonce is also the mechanism that allows the dealer to re-split the same secret: each fresh nonce produces a fresh SplitID, a fresh commitment entry, and a fresh set of shares.

PRE grants have no nonce by design (§6.2). The uniqueness domain is the grant tuple itself.

### 6.5. NFC Normalization

DIDs in the SDK may contain Unicode characters. The NFC normalization (Unicode Normalization Form C, as specified in Unicode Technical Standard #15) ensures that semantically-equivalent DID strings produce identical byte sequences. Without NFC, a DID represented as `"café"` with a composed `é` (U+00E9) would differ from `"café"` with decomposed `e + ´` (U+0065 U+0301), and the two representations would produce different SplitIDs.

The NFC normalization is applied to the DID's full string representation, not just its method-specific identifier. This is the same normalization discipline used throughout the Ortholog SDK for DID handling.

## 7. On-Log Commitment Entries

### 7.1. Two New Schemas

v7.75 introduces two schema IDs:

- `escrow-split-commitment-v1` — emitted at every escrow split.
- `pre-grant-commitment-v1` — emitted at every PRE grant.

Both schemas produce Path A commentary entries (Decision 48 discriminator): `Target_Root == null` and `Authority_Path == null`. They are structural log events covered by witness cosignatures and by the log's anchor chain, but they do not mutate SMT state.

### 7.2. Schema `escrow-split-commitment-v1`

**Payload fields:**

```json
{
  "split_id": "<32 bytes hex>",
  "dealer_did": "<NFC-normalized DID>",
  "threshold_m": <uint32>,
  "total_n": <uint32>,
  "nonce": "<32 bytes hex>",
  "commitment_set": [
    "<33 bytes hex>",
    "<33 bytes hex>",
    ...
  ]
}
```

**Field semantics:**

| Field | Type | Description |
|---|---|---|
| `split_id` | `[32]byte` | The SplitID computed per §6.1. |
| `dealer_did` | `string` | The DID of the dealer, NFC-normalized. |
| `threshold_m` | `uint32` | The threshold M. |
| `total_n` | `uint32` | The total share count N. |
| `nonce` | `[32]byte` | The random nonce used in SplitID derivation. |
| `commitment_set` | `[][33]byte` | The ordered commitment set `C_0, ..., C_{M-1}`. |

**Validation at entry admission (schema-level):**

- `threshold_m ≥ 2` and `threshold_m ≤ total_n ≤ 255`.
- `len(commitment_set) == threshold_m`.
- Each commitment is a valid compressed secp256k1 point on the curve.
- `split_id` recomputes to the expected value given `dealer_did` and `nonce`.

Note that the admission validation re-derives the SplitID to verify consistency with the dealer DID and nonce; this catches mangled entries at admission without requiring the builder to understand the Pedersen construction.

**Payload fields:**

```json
{
  "split_id": "<32 bytes hex>",
  "grantor_did": "<NFC-normalized DID>",
  "recipient_did": "<NFC-normalized DID>",
  "artifact_cid": "<32 bytes hex>",
  "threshold_m": <uint32>,
  "total_n": <uint32>,
  "commitment_set": [

```

**Field semantics:**

| Field | Type | Description |
|---|---|---|
| `split_id` | `[32]byte` | The SplitID computed per §6.2. |
| `grantor_did` | `string` | Grantor DID, NFC-normalized. |
| `recipient_did` | `string` | Recipient DID, NFC-normalized. |
| `artifact_cid` | `[32]byte` | SHA-256 content address of the artifact. |
| `threshold_m` | `uint32` | The threshold M. |
| `total_n` | `uint32` | The total share count N. |
| `commitment_set` | `[][33]byte` | The ordered commitment set. |

**Validation at entry admission:** same structural checks as the escrow schema. `split_id` re-derivation uses grantor, recipient, artifactCID — matching §6.2.

### 7.4. Uniqueness via Cryptographic Construction (Not Builder Enforcement)

Commentary entries do not mutate SMT state, so the builder cannot efficiently enforce first-write-wins at admission time — there is no leaf to consult. Uniqueness is instead enforced through two independent mechanisms:

**Construction-time uniqueness.** The SplitID construction (§6) commits to the dealer's DID plus, for escrow splits, a 32-byte random nonce. For grants, the SplitID is fully determined by the (grantor, recipient, artifact) tuple. An honest dealer will not produce two distinct commitment sets with the same SplitID because the derivation is deterministic from the dealer's inputs.

**Verifier-side equivocation detection.** If a malicious dealer publishes two commitment entries for the same SplitID, both entries exist on the log under the dealer's signature. This constitutes cryptographic proof of equivocation. Verifiers encountering two commitment entries with the same SplitID MUST:

1. Reject all shares associated with that SplitID — the dealer has demonstrated they are untrustworthy.
2. Record the equivocation evidence (both entries, their sequence numbers, their witness cosignatures) for domain-level governance action.
3. Refuse any reconstruction attempt using shares from either commitment set.

This is equivalent in security outcome to builder-side first-write-wins: a dishonest dealer's attempts to equivocate are detected and neutralized. The detection is asynchronous rather than synchronous, but the security property — no reconstruction succeeds using an equivocating dealer's shares — holds.

Honest dealers produce exactly one commitment entry per SplitID. Equivocation is evidence of malice and is handled as such by the lifecycle layer.

### 7.5. Publication Invariant

**Invariant:** Before any share is distributed to any escrow node, the escrow-split-commitment-v1 entry MUST be published to the log and covered by a witness cosignature.

**Enforcement.** `ProvisionSingleLog` and `StoreMapping` emit the commitment entry in the same log batch as the operation that produces the shares. The witness cosignature on the batch includes the commitment entry. An escrow node receiving a share without a corresponding on-log commitment entry (verified via log query) MUST reject the share.

The analogous invariant holds for PRE grants: `grantUmbralPRE` emits the `pre-grant-commitment-v1` entry atomically with the grant.

### 7.6. Why On-Log, Not Off-Log

An alternative design would publish commitments off-log, alongside the shares or in a side-channel. This is rejected because:

1. **Auditability.** On-log commitments are covered by witnesses and anchor chains. A 2050 auditor verifying a 2025 recovery can fetch the commitment entry from the log's anchored state and verify it was published at the claimed time.

2. **Single source of truth.** Off-log commitments require additional infrastructure (key-value store, replica, consistency protocol) that the log already provides.

3. **Forking scenarios.** When a log splits (scenario 3 in forking document), on-log commitments migrate automatically with the log. Off-log commitments require separate migration infrastructure.

4. **Censorship resistance.** On-log commitments are covered by witnesses and cannot be silently deleted without a detectable anchor-chain discontinuity.

The on-log publication cost is minimal: one Path A entry per split/grant, witnessed and anchored with the regular log batch.


The wire length is identical to V1 (131 bytes). V1 shares populate `BlindingFactor` and `CommitmentHash` with zeros. V2 shares populate them with the actual values. The `Version` byte distinguishes.

### 8.2. Commitment Set Serialization

For hashing (computing `CommitmentHash`) and for on-log storage, the commitment set is serialized as:

```
serialize(C_0, C_1, ..., C_{M-1}) = BE_uint32(M) || C_0 || C_1 || ... || C_{M-1}
```

where each `C_j` is the 33-byte compressed SEC 1 encoding of the corresponding secp256k1 point. The 4-byte length prefix disambiguates commitment sets of different sizes (preventing the collision where a shorter vector is a prefix of a longer one). Total length is `4 + 33 × M` bytes.

Order is significant: `C_0` corresponds to the constant term (the commitment to the secret itself), `C_1` to the linear coefficient, and so on. The verification equation depends on this ordering.

**CommitmentHash** is `SHA-256(serialize(C))`, a 32-byte value embedded in each V2 share.

Duplicate point. RKShare at offset 33 and VK at offset 66 are both defined as rk_i · G. They are the same point. One field must be dropped.

Capsule ambiguity. Offset 0 carries a "Capsule ephemeral reference (compressed)" — but the capsule's E is separately passed into PRE_VerifyCFrag as part of the verification context. Carrying E on the CFrag itself is redundant; carrying E' at offset 0 is what the field actually represents. Clarify.

Challenge-vs-commitment on wire. Draft 2's DLEQProofR on the wire requires the verifier to reconstruct e by first computing R' = z·E - e·E', which is circular. Domain applications do not need to coordinate migration — the wire format is determined by the SDK version of the grant producer.

## 9. Security Analysis

### 9.1. Threat Model

**Adversaries considered:**

- **Malicious share holder.** An escrow node or proxy that controls one or more legitimate shares or KFrags and attempts to substitute them to cause reconstruction failure or to forge CFrags.
- **Malicious coalition.** Multiple malicious share holders coordinating to jointly substitute shares or KFrags.
- **Network adversary.** Controls the transport between honest parties but cannot modify committed on-log state (witness cosignatures + anchor chain provide transport integrity).
- **Honest-but-curious dealer.** The dealer follows the protocol correctly but attempts to extract information from public commitments.
- **Equivocating dealer.** A dealer that publishes two conflicting commitment entries for the same SplitID. Detected at verifier layer per §7.4; does not enable reconstruction of a correct secret.

**Adversaries NOT considered (out of scope):**

- **Polynomial-inconsistent dealer.** A dealer that produces shares inconsistent with a published commitment. Detected at first-honest-verifier but not prevented. See §11 for the honest dealer assumption.
- **Quantum adversary.** Assumed not to exist. ECDLP on secp256k1 is assumed hard.
- **Hardware side-channel adversary.** Timing attacks, electromagnetic emanations, etc. Mitigation is a deployment-time concern (constant-time group operations in the underlying libraries) and not addressed in this ADR.

### 9.2. Security Properties Achieved

**Property 1: Substituted share detection (escrow subsystem).**

*Informal statement:* After v7.75, any substituted share is detected at verification time with overwhelming probability.

*Formal statement:* Let `C = (C_0, ..., C_{M-1})` be a commitment set produced by an honest dealer. Let `(i, f_i, r_i)` be a share consistent with `C`. Let `(i, f'_i, r'_i) ≠ (i, f_i, r_i)` be any substituted share. Then `f'_i · G + r'_i · H = ∑_j i^j · C_j` holds with probability at most `1/n` (over the randomness of the commitment set).

*Reduction:* If an adversary can produce a substituted share that verifies, the adversary can compute a collision in the commitment `Com(a, b) = a·G + b·H`. Given two distinct openings `(a, b) ≠ (a', b')` with the same commitment, the adversary can compute `log_G(H) = (a - a') / (b' - b) mod n`. This is a discrete log computation on secp256k1, which is assumed hard.

**Property 2: Substituted KFrag/CFrag detection (PRE subsystem).**

*Informal statement:* A CFrag whose `(VK_i, BK_i)` does not correspond to a valid polynomial evaluation for the committed split fails Pedersen verification with overwhelming probability.

*Reduction:* Analogous to Property 1. Breaking Pedersen binding requires computing `log_G(H)`.

**Property 3: Information-theoretic hiding of `G^{secret}`.**

*Informal statement:* The commitment set reveals nothing about `G^{secret}` beyond what computational assumptions on secp256k1 permit.

*Formal statement:* Let `(s, s') ∈ F_n × F_n` be any pair of secrets. The distributions over commitment sets `{C : C corresponds to s}` and `{C' : C' corresponds to s'}` are perfectly indistinguishable, where randomness is taken over the blinding polynomial coefficients `b_j`.

*Proof sketch:* For any commitment `C_j = a_j · G + b_j · H`, the distribution of `C_j` over uniform `b_j` is uniform on the secp256k1 subgroup. This distribution is independent of `a_j`. Therefore the commitment set is a perfect encryption of the polynomial coefficients: no adversary, bounded or unbounded, can distinguish `C(s)` from `C(s')`.

**Property 4: Soundness of DLEQ with Pedersen-absorbing transcript.**

Preserved from standard DLEQ soundness. See §5.6.

**Property 5: Cross-subsystem non-interference.**

Commitment sets published under `escrow-split-commitment-v1` and `pre-grant-commitment-v1` cannot be confused. Domain separation (§6) + schema separation (§7) provide defense in depth.

**Property 6: Cross-time non-interference.**

Shares from one split cannot be verified against a different split's commitment set. The `CommitmentHash` field in the share (hash of the commitment set) + the `SplitID` field bind the share to a specific commitment set.

**Property 7: Equivocation detection.**

If a malicious dealer publishes two commitment entries for the same SplitID, both entries exist on the log under the dealer's signature. Per §7.4, verifiers reject all shares from either commitment set. The dealer cannot use equivocation to cause reconstruction to succeed with an attacker-favorable secret.

### 9.3. Attack Scenarios Explicitly Defeated

**Scenario A: Single malicious escrow node substitutes its share.**

The node replaces `f_i` with `f'_i ≠ f_i` in its stored share. At reconstruction time, the requester receives the substituted share along with `M - 1` other (possibly honest) shares. Per Property 1, the substituted share fails Pedersen verification. Reconstruction aborts, attribution is possible (the specific failing share is identified by index).

*Comparison to v7.5:* In v7.5, reconstruction proceeds silently and produces a wrong secret. AES-GCM fails at use time with no attribution.

**Scenario B: Coalition of `M` proxies substitutes their KFrags to jointly chosen values.**

The proxies agree on an arbitrary scalar `rk'` and produce CFrags consistent with splitting `rk'`. Each CFrag passes DLEQ independently. In v7.5, Lagrange combination recovers `rk'`, and the recipient decrypts using `rk'` — which may be attacker-favorable.

*In v7.75:* Each CFrag's Pedersen check `VK_i + BK_i = ∑ i^j · C_j` fails, because the proxy-chosen `(VK_i, BK_i)` pair is inconsistent with the committed polynomial. CFrags are rejected before combination. Decryption fails; the attack is detected.

**Scenario C: Dealer publishes commitments inconsistent with the distributed shares.**

A malicious dealer publishes a commitment set `C` and produces shares that are NOT consistent with `C`. This is outside the honest dealer assumption, but the construction still detects it at first-honest-verifier time: the first escrow node (or CFrag verifier) that performs Pedersen verification sees the inconsistency and rejects its share.

*Limitation:* The dealer-malice window is [split time] to [first honest verification]. If no honest party verifies, dealer malice goes undetected. In v7.75 deployments, the publication invariant (§7.5) requires escrow nodes to verify before accepting shares, closing the window under deployment discipline.

**Scenario D: Equivocating dealer publishes two commitment entries for the same SplitID.**

The dealer publishes `(SplitID, C)` at sequence `k` and later `(SplitID, C')` at sequence `k'` with `C ≠ C'`. Per §7.4, verifiers querying the log for the SplitID discover two matching entries. Both sets of shares are rejected; no reconstruction succeeds. The dealer's equivocation is cryptographic evidence under their signature and is subject to domain-level governance sanctions.

### 9.4. Security Reductions Summary

| Attack | Reduction | Assumption |
|---|---|---|
| Share substitution | Break of Pedersen binding | ECDLP on secp256k1 |
| Dealer equivocation (committed-to-commitment) | Finding `log_G(H)` | ECDLP on secp256k1 + SHA-256 random oracle |
| `G^{secret}` leakage | Reading information from uniformly distributed commitments | Information-theoretic |
| DLEQ forgery | Break of Schnorr soundness | ROM + ECDLP on secp256k1 |
| Hash collision on SplitID / CommitmentHash | SHA-256 collision | SHA-256 collision resistance |
| Cross-implementation divergence | Transcript byte-order ambiguity | None — byte-exact spec enforces |

All reductions terminate at assumptions that are standard, well-studied, and considered secure at the 128-bit security level for classical adversaries.

## 10. Blinding Polynomial Isolation

### 10.1. The Property

Every Pedersen VSS event — every escrow split and every PRE grant — generates its own pair of independent polynomials `(f, r)`. No polynomial coefficients are shared between events. No blinding factors are reused.

### 10.2. Structural Enforcement

The `core/vss/pedersen` package's public API is designed so callers CANNOT share polynomials:

```go
// Split generates fresh polynomials internally on every call.
// There is no exported API that accepts pre-existing polynomials.
func Split(secret [32]byte, M, N int) ([]Share, Commitments, error)
```

Callers (`crypto/escrow/vss_v2.go` and `crypto/artifact/pre.go`) invoke `Split` or equivalent for each VSS event. The polynomials are internal to the `Split` invocation and are not retained beyond share emission.

### 10.3. Why This Matters

If two VSS events shared a blinding polynomial, an adversary observing both commitment sets could potentially:

1. Correlate the two commitment sets' Pedersen structure.
2. Extract the shared blinding polynomial coefficients via algebraic manipulation.
3. Use the blinding polynomial to break hiding on either event.

Independent blinding polynomials eliminate this class of attack. Each commitment set is statistically independent of every other commitment set (even for the same secret), and the hiding property is preserved per-event.

### 10.4. Auditability

The structural enforcement is visible in source:

```
core/vss/pedersen.go     → Split samples fresh randomness via crypto/rand
crypto/escrow/vss_v2.go  → SplitV2 calls Split internally
crypto/artifact/pre.go   → PRE_GenerateKFrags calls Split (or equivalent) internally
```

No function in the SDK accepts a polynomial as input. The auditor can verify this by grep:

```bash
grep -rn "polyA\|polyB\|polynomial" --include="*.go" .
```

Every call site is internal to the VSS primitive's consumers; no external API surfaces polynomial values.

### 10.5. Cross-Reuse Between Subsystems

Escrow and PRE subsystems use the same `core/vss/pedersen` primitive but call it separately. A single PRE grant's polynomial pair is never seen by the escrow subsystem, and vice versa. Structurally, they are independent events with no shared state.

## 11. Honest Dealer Assumption

### 11.1. The Assumption

v7.75 assumes the dealer (the party producing the commitment set and shares) is honest with respect to polynomial consistency. Specifically: if the dealer publishes a commitment set `C = (C_0, ..., C_{M-1})` and distributes shares `(i, f_i, r_i)` for `i ∈ {1, ..., N}`, then:

```
f_i · G + r_i · H = ∑_j i^j · C_j     for all i ∈ {1, ..., N}
```

That is, every share the dealer distributes is consistent with the commitments the dealer published.

### 11.2. Why This Is Appropriate for v7.75

In v7.75's deployment model, the dealer is always an SDK-controlled process running on infrastructure the party controls:

- **Escrow splits**: the dealer is `ProvisionSingleLog` or `StoreMapping`, running on the log operator's infrastructure.
- **PRE grants**: the dealer is `grantUmbralPRE`, running on the grantor's infrastructure.

In all these cases, the dealer is a deterministic function of the inputs (secret, threshold, nonce), implemented by the same SDK. Dealer malice would require the party to modify the SDK in a custom deployment — which is possible but represents a different class of threat (compromised infrastructure, not cryptographic attack).

### 11.3. What the Assumption Buys

Under the honest dealer assumption, Pedersen VSS's security properties simplify:

- **Share substitution is detected** against any non-dealer adversary.
- **Dealer equivocation is not possible** because the dealer commits first and reveals later; equivocation would produce on-log evidence per §7.4.
- **Distribution-time verification is meaningful**: when an escrow node receives a share, it can verify against the dealer's committed polynomial.

### 11.4. What the Assumption Does NOT Buy

- **Defense against dealer malice.** A dealer that publishes inconsistent commitments can cause reconstruction to fail in ways that are attributable to "some share is bad" but not attributable to "the dealer cheated." A malicious dealer causes denial-of-service without theft.

- **Defense against dealer-chosen polynomial.** A dealer can choose coefficients to leak information about the secret to themselves (but the dealer already knows the secret, so this is not a concern in practice).

### 11.5. When the Assumption Breaks

The honest dealer assumption fails in any scenario where the client is the dealer:

- **Client-side key splitting**: a wallet splits a user-controlled secret.
- **Delegated provisioning**: a third party produces shares for a log operator.

For these scenarios, v7.75 is not sufficient. Additional machinery (NIZK proofs of polynomial correctness, or multi-party dealer protocols with commit-reveal) is required. This is deferred to a future release.

v7.75 explicitly scopes its security guarantees to dealers running the SDK's built-in provisioning and grant code. Domain applications MUST NOT use v7.75 VSS primitives for client-generated key scenarios without additional out-of-band verification.

### 11.6. Documentation Discipline

Every function in the SDK that invokes VSS MUST document in its godoc:

- That the dealer is assumed honest.
- Which party is the dealer in the function's context.
- What failure mode occurs if the dealer is malicious.

This is a code-review enforcement, not a cryptographic one.

## 12. Distribution-Time Verification

### 12.1. The Property

An escrow node receiving a share from the dealer verifies the share against the published commitment set BEFORE storing it. If verification fails, the share is rejected and the dealer is flagged.

### 12.2. The Mechanism

`VerifyShareAgainstCommitments(share, commitments)` is callable by any party holding the share and the commitment set. Per §3.3, the function performs:

1. Version check.
2. BlindingFactor nonzero check.
3. CommitmentHash match.
4. Pedersen polynomial consistency check.

The function is pure and stateless. Distribution-time verification simply invokes it.

### 12.3. Defense-in-Depth

Distribution-time verification is defense-in-depth: even without it, reconstruction-time verification catches substitution. However, distribution-time verification offers:

- **Earlier detection.** Substitution is caught immediately, not after a recovery is triggered.
- **Attribution.** The specific node that received a bad share is identified (the receiving node discovered the failure).
- **Dealer malice detection.** The first honest escrow node reveals dealer inconsistency, bounding the dealer-malice window.

### 12.4. Deployment Integration

The `exchange/identity/mapping_escrow.go` StoreMapping flow calls `VerifyShareAgainstCommitments` on each escrow node's share before sending. Rejected shares trigger a provisioning abort.

On the receiving side (production escrow node deployments), operators MUST verify the share against the commitment entry's commitment set before acknowledging receipt. The SDK provides `VerifyShareAgainstCommitments` as a reusable primitive; integrating it into the receive path is an operator responsibility.

### 12.5. On-Wire Flow

A share arriving at an escrow node is accompanied by the SplitID (in the share envelope) and the commitment set (retrievable from the log via the SplitID). The verification is:

```
1. Extract SplitID from share envelope.
2. Fetch escrow-split-commitment-v1 entry with matching SplitID from the log.
   If multiple entries match: equivocation detected (§7.4). Reject.
3. Extract commitment_set from payload.
4. Call VerifyShareAgainstCommitments(share, commitment_set).
5. On failure: reject share, log the failure for audit.
```

## 13. Verification Cost and MSM Guidance

### 13.1. Cost Analysis

A single share verification computes:

- 1 scalar multiplication by `f_i`: `f_i · G`
- 1 scalar multiplication by `r_i`: `r_i · H`
- 1 point addition: `LHS = f_i·G + r_i·H`
- `M - 1` scalar multiplications: `(i^j mod n) · C_j` for `j ∈ {1, ..., M-1}`. (For `j = 0`, the factor is `C_0` directly, since `i^0 = 1`; no scalar mul.)
- `M - 1` point additions: `RHS = C_0 + ∑_{j=1}^{M-1} (i^j mod n) · C_j`
- 1 point equality check: `LHS == RHS`

Total: `M + 1` scalar multiplications and `M` point additions, plus one equality check.

At typical secp256k1 timings (~100µs per scalar multiplication on modern x86_64), this is approximately `(M + 1) × 100µs` milliseconds.

For common thresholds:

| M | Cost per share |
|---|---|
| 3 | ~0.4 ms |
| 5 | ~0.6 ms |
| 10 | ~1.1 ms |

For reconstruction with `M` shares, total verification cost is `M × (M + 1) × 100µs`. At M=5, this is `5 × 6 × 100µs = 3.0 ms`, which is negligible compared to log fetch latency.

### 13.2. MSM (Multi-Scalar Multiplication) Guidance

For large thresholds or batch verification scenarios, multi-scalar multiplication (MSM) is more efficient than sequential scalar multiplications. An MSM computes `∑ c_j · P_j` in one pass using Pippenger's algorithm or similar, achieving sub-linear cost in the number of scalars.

**Recommendation:** For `M ≥ 10`, use MSM for the RHS computation `∑ (i^j mod n) · C_j`. The `decred/dcrd/dcrec/secp256k1/v4` library provides primitive operations; an MSM implementation can be added as an optimization if operational scale demands it.

**Recommendation:** For `M < 10`, naive sequential computation is optimal or near-optimal. MSM overhead (Pippenger setup) exceeds savings.

**Recommendation for v7.75:** Use naive sequential verification throughout. MSM optimization is deferred to v8 if profiling identifies verification as a bottleneck.

### 13.3. Batch Verification Across Multiple Shares

A common scenario: at reconstruction time, verify all `M` shares in one pass. For each share `i`, the RHS `∑ (i^j mod n) · C_j` is independent; verification cannot be batched naively.

**Optimization:** Pre-compute `i^j mod n` for all `i` and `j` used. This is cheap arithmetic and shared across shares.

**Optimization:** Pre-compute `(i^j mod n) · C_j` for small `i^j` values if the threshold is small. Memoization reduces redundant scalar multiplication.

These optimizations reduce the constant factor but do not change the asymptotic cost.

## 14. Seed Lifecycle

### 14.1. The Seed Is Permanent

The seed `"ortholog/core/vss/pedersen/h-generator/v1"` is **permanent** for v7.75 and all subsequent releases that do NOT change the generator derivation. The `v1` suffix specifically tracks the derivation construction, not the release version.

### 14.2. When the Seed Changes

A seed version bump (`v1` → `v2`) occurs if and only if:

1. The try-and-increment procedure changes (e.g., moving to a different hash function, or to an SSWU map).
2. The seed string itself changes (e.g., reorganization of the package path).
3. The security properties of the existing derivation are compromised (e.g., a secp256k1 weakness is discovered that affects the uniform-distribution argument).

A seed bump produces a new generator `H_v2`. Existing commitments under `H_v1` are NOT retroactively invalidated — they remain verifiable using `H_v1`. New commitments use `H_v2`.

Effectively, a seed bump is equivalent to introducing a new Pedersen variant. Both variants coexist; commitments identify their variant via the seed version used at creation.

### 14.3. Version Tracking in Wire Format

The Pedersen variant (seed version) is identified by the containing protocol version, not by the commitment itself. A v7.75 commitment uses `H_v1`. A future v8 commitment that bumps the seed uses `H_v2`.

Verifiers determine which `H` to use based on the protocol version of the commitment entry (from the Control Header). This is checked at the verifier's parameter resolution step.

### 14.4. Release vs Seed Version

The Ortholog SDK has two independent versions for the generator:

- **Release version** (v7.75, v8, v8.1, ...): tracks overall SDK releases.
- **Seed version** (v1, v2, ...): tracks only changes to the generator derivation.

For v7.75 through any v7.x that does not change the derivation, the seed remains `v1`. If v8 changes the derivation, the seed moves to `v2`. If v8 is purely additive in other ways and doesn't touch the derivation, the seed stays `v1`.

This discipline prevents unnecessary seed churn. The release version is user-facing; the seed version is cryptographic-protocol-facing.

## 15. Alternatives Considered

### 15.1. Feldman VSS

**Pros:** Simpler construction (one commitment per coefficient), slightly smaller wire format (no blinding polynomial).

**Cons:** Reveals `G^{secret}` publicly. For escrow (secret is a symmetric key) and PRE (secret is `rk`), `G^{secret}` is sensitive.

**Verdict:** Rejected. Hiding loss is unacceptable for Ortholog's use cases.

### 15.2. Hash-Binding

**Construction:** Instead of Pedersen, bind each share to the dealer's commitment via `SHA-256(share || dealerKey)`. Verification: recompute hash.

**Pros:** No new cryptographic primitive. Uses only SHA-256.

**Cons:** Fails open — the verifier can compute the hash only if it has the share. A substituted share still produces a valid hash because the substitution changes the hash's input. The verifier cannot distinguish "valid share" from "substituted share" without the polynomial context.

**Verdict:** Rejected after self-correction during design review. The earlier plan incorrectly proposed this; cryptographic analysis showed it does not provide the claimed property.

### 15.3. NIZK-Backed VSS

**Pros:** Defends against adversarial dealers. Stronger security.

**Cons:** Proof sizes of several KB per share. Verification times in tens of milliseconds. Implementation complexity significantly higher than Pedersen.

**Verdict:** Deferred. The honest dealer assumption is appropriate for v7.75; NIZKs are deferred to a future release that explicitly addresses client-generated-key scenarios.

### 15.4. Publicly Verifiable Secret Sharing (PVSS)

**Pros:** Enables any public verifier to check share validity, not just share holders. Useful for recovery scenarios where the recovering party doesn't hold a share.

**Cons:** Heavier construction. Pedersen VSS already supports public verification (given the commitment set, any party can verify a share) at lower cost.

**Verdict:** Pedersen is a form of PVSS for the verification property; full PVSS constructions add properties (encrypted shares, public distribution) that v7.75 does not need.

### 15.5. RFC 9380 SSWU Hash-to-Curve for H Derivation

**Pros:** Constant-time, statistically unbiased, standards-aligned.

**Cons:** ~200 lines of SSWU implementation the auditor must review. Adds complexity for properties (timing side-channel resistance, per-invocation unbiasedness) that are irrelevant to a one-time startup derivation. No Go library exposes secp256k1 SSWU without pulling in a larger dependency.

**Verdict:** Rejected in favor of try-and-increment. Bitcoin Core, Zcash Sapling, and Monero all use try-and-increment for analogous Pedersen generator derivation. ADR-005 Draft 1 incorrectly specified SSWU; Draft 2 aligns with the implementation.

### 15.6. Alternative Curves

- **BLS12-381**: pairing-friendly, used for BLS signatures elsewhere in the SDK. Pedersen on G1 would work, but introduces two curves in the VSS subsystem for no gain.
- **Ristretto/Curve25519**: excellent performance, but the SDK's secp256k1 existing infrastructure (ECIES, Umbral, entry signing) makes secp256k1 the natural choice.

**Verdict:** secp256k1. Consistency with the rest of the SDK, no pairing needed for Pedersen, proven library support via `decred/dcrd/dcrec/secp256k1/v4`.

### 15.7. Off-Log Commitment Distribution

**Construction:** Publish commitments in a side channel (CAS, DHT, key-value store) rather than on-log.

**Pros:** Smaller on-log footprint. Log pressure reduced.

**Cons:** Requires separate infrastructure for commitment availability. Not covered by witness cosignatures or anchor chain. Loses auditability and forking-compatibility.

**Verdict:** Rejected. On-log is the consistent infrastructure pattern.

### 15.8. Single Schema for Both Escrow and PRE

**Construction:** One schema `vss-commitment-v1` with a "subsystem" field distinguishing escrow from PRE.

**Pros:** Fewer schemas. Simpler schema registry.

**Cons:** Conflates two different subsystems' lifecycle. Domain separation is weakened. Schema evolution becomes coupled across subsystems.

**Verdict:** Rejected. Two schemas preserve cleaner separation. The overhead of two schema definitions is minimal.

### 15.9. Random-Nonce PRE Grant SplitID

**Construction:** `SplitID_grant = SHA-256(DST || grantor || recipient || artifactCID || nonce || timestamp)`.

**Cons:** Nonce is unknown to recipient and proxies at verification time. They cannot reconstruct the SplitID to look up the commitment entry. Either the SDK must thread the nonce through every CFrag and every grant notification, or it must publish a side-index that maps (grantor, recipient, artifactCID) → SplitID. Both are extra infrastructure for no security gain.

**Verdict:** Rejected. Determinism from the grant tuple eliminates discoverability problems and is not a security loss (the tuple is already unique per legitimate grant; re-grants rotate the artifactCID per §6.2.1).

## 16. Out of Scope

### 16.1. Post-Quantum Security

v7.75's security rests on the discrete log assumption on secp256k1, which does not hold against quantum adversaries. Post-quantum migration is a protocol-level effort scheduled for a future release (v9 or beyond), likely involving:

- Lattice-based commitments (Kyber, Dilithium-family).
- Hash-based threshold signatures (SPHINCS+-family adaptations).
- New wire formats, new DSTs, new schemas.

### 16.2. Client-Generated-Key Dealer Malice

The honest dealer assumption (§11) is load-bearing. For scenarios where the client is the dealer, v7.75 provides no defense against:

- Dealer committing to inconsistent polynomials.
- Dealer leaking shares that don't match commitments.

Future work: NIZK proofs of polynomial correctness, or multi-party dealer protocols.

### 16.3. Pedersen Variants Beyond v7.75

v7.75 commits to a specific Pedersen construction. Alternative variants (e.g., Pedersen over different subgroups, Pedersen with compressed commitments via Bulletproofs) are not considered.

### 16.4. MSM Optimization

Multi-scalar multiplication for batch verification is noted in §13.2 as future work. Not in v7.75.

### 16.5. Share Refresh

Proactive secret sharing (periodic refresh of shares without changing the underlying secret) is not supported by v7.75. To refresh shares, issue a new split with the same secret and distribute new shares. This produces a new commitment entry and new SplitID.

### 16.6. Threshold Decryption

v7.75 is about threshold *secret sharing* (reconstructing a secret from shares). Threshold *decryption* (a set of parties jointly decrypting without any party learning the secret) is a related but distinct primitive. The Umbral PRE subsystem provides a form of threshold decryption via the CFrag mechanism; v7.75 strengthens that mechanism's binding but does not generalize to other threshold decryption schemes.

### 16.7. Witness Coverage Proof

The on-log commitment entries are covered by witness cosignatures as part of the normal log publication flow. This ADR does not introduce new witness requirements; it reuses existing witness infrastructure. Any change to witness coverage is scope for a different ADR.

## 18. Consequences

### 18.1. Positive

- **Substitution attacks defeated.** Both escrow and PRE subsystems gain cryptographic substitution detection.
- **Audit-grade cryptographic foundation.** Every primitive runs on production-audited libraries (gnark-crypto, decred-secp256k1-v4, Go stdlib). External audit of ADR-005 plus `core/vss` is achievable scope.
- **Forking compatibility.** County splits and specialty court scenarios (forking document scenarios 2B, 3) gain cryptographic verification of escrow operations across jurisdictional transfers.
- **No domain action required.** v7.75 is additive at the API level. Domain applications benefit without modification.
- **Reproducibility.** ADR-005 plus test vectors enable future Rust/TypeScript implementations to interoperate byte-exactly.

### 18.2. Negative

- **Wire format growth.** CFrags grow from 163 to 196 bytes (+20%). Network and storage overhead in PRE flows.
- **Verification cost.** CFrag verification adds Pedersen check (~0.5ms per CFrag); share verification adds Pedersen check (~0.6ms per share at M=5).
- **Commitment entry log pressure.** Each escrow split and PRE grant produces one additional log entry. Witness and anchor costs increase marginally.
- **Honest dealer assumption.** v7.75 scope does not include client-generated key dealer malice. Documentation discipline is required to prevent misuse.
- **PRE grant re-issuance requires artifact rotation.** Deterministic SplitID means a new grant to the same (grantor, recipient, artifact) tuple requires rotating the artifact (§6.2.1). This is by design but changes workflow for domain applications that previously assumed re-grant was free.

### 18.3. Neutral

- **V1 share format remains supported.** Existing deployments continue to work. v7.75 is a floor, not a ceiling.
- **Protocol version unchanged.** v7.75 stays at protocol version 7. The commitment schemas are schema-layer additions.

### 18.4. Migration Impact

**For SDK consumers:** None. Existing v7.5 code compiles and runs against v7.75.

**For new grants/splits in v7.75+:** V2 becomes default. Commitment entries emitted automatically.

**For legacy v7.5 shares:** Remain valid, remain reconstructible. No forced migration.

**For domain applications:** Optional upgrade to V2 by reissuing splits/grants. No breaking change at the API surface.

### 18.5. Forking Scenario Impact

- **Scenario 2B (heavyweight specialty court):** every new escrow split emits a commitment entry covered by witnesses. A 2050 auditor in a different jurisdiction can verify share integrity without trusting the originating court's infrastructure.
- **Scenario 3 (county split):** mapping-escrow recovery at new jurisdiction becomes cryptographically verifiable. Substituted shares from the splitting county are detected at the new county's reconstruction.
- **Other scenarios (1, 2A, 4, 5, 6):** unaffected. v7.75 is transparent to them.

## Appendix B — Proof Sketches

### B.1. Pedersen Binding Reduces to ECDLP

**Claim:** If an adversary can produce two distinct openings `(a, b) ≠ (a', b')` with `a·G + b·H = a'·G + b'·H`, the adversary can compute `log_G(H) = (a - a') / (b' - b) mod n`.

**Proof:** From `a·G + b·H = a'·G + b'·H`, we get `(a - a')·G = (b' - b)·H`. If `b ≠ b'`, we can compute `H = ((a-a')·(b'-b)^{-1}) · G` (inverse of `b'-b` exists in `F_n` since `n` is prime). Therefore `log_G(H) = (a-a') · (b'-b)^{-1} mod n`, which is ECDLP on secp256k1.

If `b = b'`, then `(a-a')·G = 0`, so `a ≡ a' mod n`, contradicting the assumption that the openings are distinct.

### B.2. Pedersen Hiding Is Information-Theoretic

**Claim:** For any secret `s`, the commitment `C = s·G + b·H` with uniform `b ∈ F_n` is uniformly distributed on the secp256k1 subgroup.

**Proof:** The map `b ↦ b·H` is a bijection from `F_n` to the secp256k1 subgroup (since `n` is prime and `H` is a generator). Adding the constant `s·G` is also a bijection (translation on an abelian group). Therefore `b ↦ s·G + b·H` is a bijection from `F_n` to the subgroup. Uniform `b` produces uniform `C`.

**Corollary:** `C` reveals no information about `s` in the information-theoretic sense. Any pair of secrets `(s, s')` produces identical distributions over commitments.

### B.3. DLEQ + Pedersen Non-Interference

**Claim:** Absorbing `BK_i` and commitment set `C` into the DLEQ Fiat-Shamir challenge does not weaken DLEQ soundness.

**Proof sketch:** The standard Fiat-Shamir transform derives the challenge as `e = Hash(statement || commitments)` where `Hash` is modeled as a random oracle. Soundness holds for any random oracle inputs; adding more inputs (such as `BK_i`, `C`) produces a different random oracle but does not weaken the soundness argument.

Concretely, an adversary forging a DLEQ proof for statement `(G, E, VK_i, E')` with challenge `e = Hash(..., BK_i, C, ...)` cannot reuse the same proof for `(G, E, VK_i, E')` with a different `(BK'_i, C')` because the challenge differs. The Pedersen-absorbing hash is still a random oracle by the same argument.

The extraction argument for Schnorr-style DLEQ proofs works unchanged: from two accepting transcripts with different challenges, one extracts `rk_i`. Whether the challenges come from a hash of the statement alone or from a hash that includes `BK_i` and `C`, the extraction is the same.

### B.4. Blinding Polynomial Isolation Prevents Cross-Event Attacks

**Claim:** If two splits use independent random blinding polynomials, observing both commitment sets reveals no correlation beyond what is revealed by the secret polynomials alone.

**Proof sketch:** Let `C(s_1, r_1)` and `C'(s_2, r_2)` be commitment sets for two splits with independent blinding polynomials `r_1` and `r_2`. By Appendix B.2, each is uniformly distributed. Independence of `r_1` and `r_2` means the joint distribution `(C, C')` is uniform on the product of subgroups.

Any correlation between `s_1` and `s_2` is hidden by the independent blinding. The adversary observes only uniformly random commitments whose distributions do not depend on the secrets.

### B.5. Honest Generator Derivation

**Claim:** Under the random oracle model for SHA-256, the discrete log `log_G(H)` is uniformly distributed in `F_n` and uncomputable in polynomial time.

**Argument (structural):**

1. The seed string is a public constant committed to source control before any test vector was computed. An author intending to produce a known-discrete-log `H` would need to find a seed that, hashed against some counter, yields a specific x-coordinate. This is second-preimage resistance on SHA-256.

2. The try-and-increment procedure has no input other than the seed and the counter. It does not reference `G`'s coordinates. Therefore the derivation of `H` is structurally independent of `G`.

3. Under the random oracle model, each candidate x is uniform in `F_p`. Conditioned on the candidate yielding an on-curve y, the resulting `(x, y)` is uniform on the curve. Since secp256k1 has prime order, uniform on the curve is equivalent to `k · G` for uniform `k ∈ F_n`.

4. Computing `k` from `k · G` is ECDLP on secp256k1, assumed hard.

The argument depends on the seed genuinely being fixed before the coordinates were consulted. In the Ortholog source tree, this is verifiable from git history: the seed was committed in the same patch as the derivation code, prior to any test vector generation.
