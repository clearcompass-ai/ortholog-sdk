Ortholog SDK Identity & Cryptographic Architecture (Expanded)The Ortholog SDK categorizes identity into three distinct paradigms. The cryptographic boundary between these paradigms is fiercely protected, ensuring that a signature intended for a Web3 wallet cannot be maliciously routed to an institutional public key.Paradigm 1: Entity / Institutional IDs (did:web)The Concept: Identity is tied to a DNS domain (e.g., did:web:davidson.court.gov#clerk-1). The institution hosts a DIDDocument that lists the public keys authorized to act on its behalf.Signature Verification: Handled by did/web_verifier.go. The verifier strictly requires a fragment match (or a single-key document) to avoid ambiguous "try every key" routing. It dispatches standard curve signatures (ECDSA, Ed25519) directly to their respective primitives.Recovery / Rotation Strategy (In-Place Succession): The identity outlives the cryptographic key. If a key is compromised, the institution updates their hosted did.json file. The log identity remains intact, historical signatures remain valid, and the new key seamlessly takes over. Escrow nodes are used to back up these institutional keys.Paradigm 2: Individual Web3 IDs (did:pkh)The Concept: The identity is the cryptography. It uses a CAIP-10 blockchain account address (e.g., did:pkh:eip155:1:0xAbCd...), representing a user operating a standard Web3 wallet like MetaMask.Signature Verification (The Ecrecover Pipeline): did/pkh.go parses the string and synthesizes a document with VerificationMethodSecp256k1Recovery. The verification relies on the Ethereum primitives provided in crypto/signatures/ethereum_primitives.go:Digest Generation: The SDK supports both EIP191Digest (the \x19Ethereum Signed Message:\n standard used by personal_sign) and EntrySigningDigest (the EIP-712 typed data standard).Recovery: The SDK uses Decred's RecoverSecp256k1 to extract the 65-byte uncompressed public key from the 65-byte signature (r || s || v).Address Binding: It drops the 0x04 prefix, computes the legacy Keccak256 hash, and takes the last 20 bytes (AddressFromPubkey). If this matches the CAIP-10 address, the signature is valid.Recovery / Rotation Strategy (Mapping Escrow Transition): A Web3 address cannot be separated from its private key. If the seed phrase is lost, the did:pkh is permanently dead. Recovery requires proving the user's real-world identity via exchange/identity/mapping_escrow.go, after which network admins publish a Succession Entry migrating all systemic authority to a brand new did:pkh wallet.Paradigm 3: Self-Certifying Cryptographic IDs (did:key)The Concept: Identity is the raw public key itself, encoded in a multicodec string (e.g., did:key:z6Mk...).Signature Verification: Handled strictly by did/key_verifier.go. It explicitly REJECTS EIP-191 and EIP-712 algorithms. did:key has no address, only a pubkey, so mixing address-based recovery with did:key is a fatal category error.Recovery / Rotation Strategy: None. A did:key is ephemeral. If the key is lost, the identity ceases to exist. There is no escrow or succession.Paradigm 4: The Consensus Layer (BLS Witnessing)The Concept: While origin entries are signed by the identities above, the transparency log's state is cosigned by a decentralized network of witnesses using BLS12-381 cryptography (crypto/signatures/bls_verifier.go).Optimistic Aggregation: To scale nationally, the verifier computes a single $O(1)$ pairing check regardless of witness count: $e(\Sigma sigs, G2_{gen}) == e(HashToG1, \Sigma pubkeys)$. If it fails, it gracefully falls back to $O(N)$ individual checks for exact attribution.The Rogue Key Defense: This aggregate math is only secure if an attacker cannot manipulate the public key pool. The SDK implements VerifyBLSPoP (Proof-of-Possession). Witness registrars must verify this PoP before admitting a key to the network, mathematically destroying the rogue-key attack vector.
