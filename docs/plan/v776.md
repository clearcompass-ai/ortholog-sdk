# Phase D — v7.76 Complete Execution Plan

Seven groups. Dependencies flow forward: Group N assumes 1..N-1 are green. Every group has a single named closure proof. v7.76 is scoped to ship as a complete SDK release; the four schemas, the lifecycle functions, the manifest escrow primitive, and the schema fingerprint discipline land together.

The plan assumes v7.75 (`adr.md`, ADR-005) is fully implemented and tagged `v7.75.0-phase-c`. Every primitive referenced below — `LengthPrefixed`, `ComputeEscrowSplitID`, `SplitV2`, `Reconstruct`, `EncryptForNode`, the mutation-audit runner at `cmd/audit-v775`, the universal length-prefix rule, the canonical SplitID constructions — is shipped, tested, and locked. Phase D extends this foundation; it does not modify it.

The plan also assumes ADR-006 (`adr-006.md`) is locked and authoritative. Every schema field, every commitment construction, every algorithm discriminator referenced below is specified there.

## Group 1 — Schema Fingerprint Discipline

**Packages:** `core/envelope/`, `crypto/hash.go` (read-only reference), `schema/` (read-only reference)

**Purpose:** Ship the foundation that makes every other v7.76 schema work. The fingerprint binding closes the gap where `SchemaRef` pins a schema by log position but the verifier has no on-log proof that the schema bytes at that position are what the entry was signed against. Every other v7.76 group depends on this primitive — content-commitment-v1 entries pin fingerprints; derivation-binding-v1 entries verify both pinned fingerprints match; archival-manifest-v1 entries pin fingerprints redundantly; authorization-grant-v1 entries pin fingerprints to lock authorization context to schema state at the time of authorization.

**Subgroup 1.1 — `SchemaFingerprint` field on `ControlHeader`.** Extend `core/envelope/envelope.go`'s `ControlHeader` struct with a new field:

```go
SchemaFingerprint *[32]byte `json:"schema_fingerprint,omitempty"`
```

The pointer-with-omitempty pattern preserves v7.75 backward compatibility: entries without the field are existing v7.75 entries with `nil` fingerprint; entries with the field are v7.76+. JSON serialization omits `nil` fingerprints. Canonical bytes serialization (per v7.75 `core/envelope/serialize.go` discipline) includes the fingerprint when non-nil and excludes the field entirely when nil — this is the same Option A discipline established for `SchemaParameters` in v7.5/v7.75 except applied at the envelope rather than the schema layer.

The fingerprint is required (non-nil) on every entry that carries `SchemaRef`. The envelope validator `ValidateControlHeader` enforces this: any v7.76+ entry with `SchemaRef != nil && SchemaFingerprint == nil` is rejected at construction time as malformed. This makes "an entry references a schema but doesn't pin its fingerprint" structurally impossible.

**Subgroup 1.2 — `ComputeSchemaFingerprint` primitive.** New function in a new file `schema/fingerprint.go`:

```go
func ComputeSchemaFingerprint(schemaEntry *envelope.Entry) ([32]byte, error)
```

The function calls `envelope.SerializeCanonical(schemaEntry)` to obtain the schema entry's canonical bytes (the same byte sequence that was signed and hashed for the entry's identity), then computes `LengthPrefixed("ORTHOLOG-V7.76-SCHEMA-FINGERPRINT-v1", canonical_bytes)`. Returns the 32-byte SHA-256 digest.

The DST is versioned (`-v1`) per ADR-006 §1 to allow future migrations (e.g., to a ZK-friendly hash) without breaking the rule itself. Future versions add new functions (`ComputeSchemaFingerprintV2`) with new DSTs; v7.76 ships only V1.

The fingerprint is computed over the *canonical bytes* of the entire schema entry, not just the schema's parameters payload. This binds the fingerprint to the full envelope-level context — signer DID, destination, event time, control header, schema parameters — so a schema entry republished at a different log position with the same parameters but different envelope context has a different fingerprint. This is by design: the fingerprint identifies a specific schema *publication event*, not just its parameter content.

**Subgroup 1.3 — Verifier integration.** Extend `core/envelope/verify.go` (read into the verifier's chain-of-trust check) with a new function:

```go
func VerifySchemaFingerprint(entry *envelope.Entry, fetcher types.EntryFetcher) error
```

The function: extracts `SchemaRef` and `SchemaFingerprint` from the entry's control header; if either is nil and the other is non-nil, returns a structured "fingerprint discipline violation" error; if both are non-nil, fetches the schema entry at `SchemaRef`, computes its fingerprint via `ComputeSchemaFingerprint`, compares to the pinned fingerprint, returns mismatch error if they differ. Two new mutation switches gate this verification: `muEnableSchemaFingerprintRequired` (rejects entries with `SchemaRef != nil && SchemaFingerprint == nil`) and `muEnableSchemaFingerprintMatch` (rejects entries where the recomputed fingerprint differs from the pinned one).

The verifier is called from every entry verification path that already verifies `SchemaRef`: `VerifyEntryStructure`, `VerifyAdmission`, the verifier-layer authority evaluator. The integration is additive — existing checks remain in place, and the fingerprint check is a new gate that runs after the existing schema-ref-existence check.

**Subgroup 1.4 — Builder integration.** Every v7.75 builder that accepts a `SchemaRef` parameter (`BuildRootEntity`, `BuildAmendment`, `BuildScopeCreation`, `BuildScopeAmendment`, `BuildEnforcement`, `BuildKeyRotation`, `BuildKeyPrecommit`, `BuildPathBEntry`, plus the v7.75 `BuildPREGrantCommitmentEntry` and `BuildEscrowSplitCommitmentEntry`) is extended to require a fetcher parameter for fingerprint computation. The fetcher resolves the schema entry; the builder computes its fingerprint via `ComputeSchemaFingerprint` and pins it on the constructed entry's control header.

This is a breaking API change for builders that accept `SchemaRef`. The migration path: builders gain a new optional `Fetcher` field on their params struct. When `SchemaRef != nil`, `Fetcher` MUST be non-nil; the builder returns an error otherwise. When `SchemaRef == nil`, `Fetcher` is ignored. Callers migrating from v7.75 to v7.76 update their builder invocations to supply a fetcher when they supply a schema ref. The `validateCommon` helper in `builder/entry_builders.go` is extended with a `validateSchemaRefAndFingerprint(schemaRef, fetcher)` call that enforces the conjunction.

**Closure proof.**

- `TestSchemaFingerprint_GoldenVector` pins the fingerprint computation against a fixture schema entry at `schema/testdata/fingerprint_vector.json` containing a complete schema entry and its expected 32-byte fingerprint.

- `TestSchemaFingerprint_DSTBoundary` validates that two different DSTs (e.g., `"ORTHOLOG-V7.76-SCHEMA-FINGERPRINT-v1"` vs `"ORTHOLOG-V7.76-SCHEMA-FINGERPRINT-v2"`) produce different fingerprints over identical canonical bytes — confirms versioning is structurally enforced.

- `TestVerifySchemaFingerprint_MissingFingerprint_Rejected` constructs a v7.76+ entry with `SchemaRef` set and `SchemaFingerprint` nil, calls `VerifySchemaFingerprint`, asserts rejection.

- `TestVerifySchemaFingerprint_TamperedSchema_Rejected` constructs a valid v7.76 entry with correct fingerprint, then tampers with the schema entry's bytes at the referenced position (e.g., flips a single byte in the parameters payload), confirms verification fails with mismatch error.

- `TestBuilder_RejectsSchemaRefWithoutFetcher` validates each affected builder rejects construction when `SchemaRef != nil && Fetcher == nil`.

- Two binding tests for the new mutation switches: `TestMuEnableSchemaFingerprintRequired_Binding` flips the switch off, confirms entries without fingerprints are accepted (proving the gate is load-bearing). `TestMuEnableSchemaFingerprintMatch_Binding` flips the switch off, confirms entries with mismatched fingerprints are accepted.

- `core/envelope/fingerprint.mutation-audit.yaml` registry registered with the Group 4 audit runner.

**Verification for Group 1.** Schema fingerprint discipline is enforced at builder, envelope, and verifier layers. Mutation-audit registry passes. `grep -rn 'SchemaRef' --include='*.go'` reviewed across builder and verifier code; every site that processes `SchemaRef` either computes or verifies the fingerprint. `go test ./core/envelope/... ./schema/... ./builder/...` green. Golden vector locked.

## Group 2 — Content Commitment Primitive

**Packages:** `crypto/artifact/` (none), `schema/`, `lifecycle/`, `crypto/hash.go` (read-only reference)

**Purpose:** Ship the foundational v7.76 commitment primitive. Content commitments are the substrate every other v7.76 schema builds on — archival manifests reference content commitments for masters; derivation bindings link content commitments together; authorization grants are bound to content commitments through subject pointers. The commitment primitive must be airtight before any of these higher-level compositions are wired up.

**Subgroup 2.1 — `ContentCommitment` struct and serializers.** New types in a new file `schema/content_commitment.go`:

```go
type ContentCommitmentPurpose uint8

const (
    PurposeMaster        ContentCommitmentPurpose = 1
    PurposeDerived       ContentCommitmentPurpose = 2
    PurposeAuditSnapshot ContentCommitmentPurpose = 3
)

type ContentCommitment struct {
    SubjectPointer    types.LogPosition
    SchemaRef         types.LogPosition
    SchemaFingerprint [32]byte
    CommitmentValue   CommitmentValue
    Purpose           ContentCommitmentPurpose
}

type CommitmentValue struct {
    Algorithm uint8 // 0x01 = SHA-256 via LengthPrefixed
    Digest    []byte // length depends on algorithm; 32 bytes for SHA-256
}
```

The `Purpose` enum starts at 1 (not 0) so the zero value is distinguishable as "unclassified" — this follows the ADR-005 §7 `ConstraintState` pattern of making the zero value structurally meaningful. A `Purpose: 0` content commitment is rejected at admission as malformed.

`SerializeContentCommitment` produces a length-prefixed payload: `SubjectPointer(serialized) || SchemaRef(serialized) || SchemaFingerprint || Algorithm(1 byte) || Digest(length-prefixed) || Purpose(1 byte)`. Each variable-length component is length-prefixed per the universal v7.75 rule. `DeserializeContentCommitment` validates field-by-field: well-formed log positions, exactly 32-byte fingerprint, recognized algorithm discriminator, digest length matches algorithm (32 bytes for SHA-256), recognized purpose value.

**Subgroup 2.2 — `ComputeContentCommitmentDigest` primitive.** New function in `schema/content_commitment.go`:

```go
func ComputeContentCommitmentDigest(
    schemaRef types.LogPosition,
    serializedBytes []byte,
) [32]byte
```

The function calls `LengthPrefixed("ORTHOLOG-V7.76-CONTENT-COMMITMENT-v1", schemaRefBytes(schemaRef), serializedBytes)`. Returns the 32-byte SHA-256 digest. The DST is versioned per ADR-006 §2.

`schemaRefBytes` is a helper that produces the canonical byte representation of a `LogPosition`: 2-byte big-endian length of LogDID + LogDID UTF-8 bytes + 8-byte big-endian Sequence. This canonicalization is locked in `types/log_position.go` with a `CanonicalBytes()` method; the same canonicalization is used wherever a LogPosition appears as a field in a hash input.

**Subgroup 2.3 — `content-commitment-v1` schema registration.** New file `schema/content_commitment_v1.go` declares the schema:

```go
const ContentCommitmentSchemaID = "ortholog/content-commitment-v1"

func RegisterContentCommitmentSchema(registry *SchemaRegistry) error
```

The schema declares: payload validator (calls `DeserializeContentCommitment`, then validates the commitment via `VerifyContentCommitment`), well-known parameter set (the schema is parameterless — the commitment data lives in the entry's domain payload, not in schema parameters), and the admission-time fingerprint check (calls `VerifySchemaFingerprint` from Group 1).

The schema entry itself is built once by a schema authority (typically the SDK's bootstrap initialization or a domain application's onboarding script) and committed to a known log position. Every domain application that uses content commitments references this single canonical schema entry. The fingerprint discipline ensures all references are bound to the same schema bytes.

**Subgroup 2.4 — `BuildContentCommitmentEntry` builder.** New function in `builder/entry_builders.go`:

```go
type ContentCommitmentEntryParams struct {
    Destination       string
    SignerDID         string
    Fetcher           types.EntryFetcher
    Commitment        ContentCommitment
    EventTime         int64
}

func BuildContentCommitmentEntry(
    params ContentCommitmentEntryParams,
) (*envelope.Entry, error)
```

Calls `validateCommon(params.SignerDID, params.Destination)`. Verifies the commitment is well-formed: subject pointer non-zero, schema ref non-zero, fingerprint non-zero, digest length matches algorithm, purpose is recognized. Computes the schema fingerprint for the content-commitment-v1 schema itself via the fetcher. Serializes the commitment and constructs a Path A commentary entry with the serialized commitment as `DomainPayload` and the schema fingerprint pinned on the control header. Returns the unsigned entry.

**Subgroup 2.5 — `VerifyContentCommitment` verifier.** New function in `schema/content_commitment.go`:

```go
func VerifyContentCommitment(c *ContentCommitment) error
```

Verifies four properties, each gated by its own mutation switch:

1. `muEnableContentCommitmentSubjectPointerNonZero` — `SubjectPointer` is well-formed (non-empty LogDID, valid sequence).
2. `muEnableContentCommitmentSchemaRefNonZero` — `SchemaRef` is well-formed.
3. `muEnableContentCommitmentFingerprintNonZero` — `SchemaFingerprint` is non-zero (i.e., not the all-zeros sentinel).
4. `muEnableContentCommitmentPurposeValid` — `Purpose` is one of `PurposeMaster`, `PurposeDerived`, `PurposeAuditSnapshot`.

The verifier explicitly does NOT verify the digest matches any particular plaintext — that verification requires the plaintext as input and is performed by audit-time tooling, not by admission. The on-log commitment is the pinned claim; verification of the claim against actual content is a separate operation.

**Subgroup 2.6 — `FetchContentCommitment` lookup primitive.** New function in `schema/content_commitment.go`:

```go
func FetchContentCommitment(
    fetcher types.EntryFetcher,
    pointer types.LogPosition,
) (*ContentCommitment, error)
```

Fetches the entry at the given log position, validates it is a content-commitment-v1 entry (correct schema ref), deserializes the commitment, returns the structured value. Used by every higher-level v7.76 lifecycle function that needs to follow a `*_commitment_pointer` field.

**Closure proof.**

- `TestContentCommitment_GoldenVector` pins a fixture commitment at `schema/testdata/content_commitment_vector.json` with a known subject pointer, schema ref, fingerprint, value, and purpose, alongside the expected serialized bytes.

- `TestComputeContentCommitmentDigest_GoldenVector` pins the digest computation: given a fixture schema ref and serialized bytes, the digest matches the locked 32-byte expected value.

- `TestComputeContentCommitmentDigest_DSTBinding` validates that two different schema refs produce different digests over identical content, confirming the schema ref is structurally bound into the digest.

- Four binding tests for the four mutation switches, each constructing a malformed commitment that the corresponding gate rejects.

- `TestBuildContentCommitmentEntry_HappyPath` builds a complete entry, validates the resulting envelope structure, signature placeholder, and serialized payload.

- `TestBuildContentCommitmentEntry_RejectsMissingFetcher` validates the builder rejects construction without a fetcher.

- `TestFetchContentCommitment_HappyPath` fetches a previously-built entry and confirms deserialization round-trips.

- `TestFetchContentCommitment_WrongSchema_Rejected` validates the lookup rejects entries that aren't content-commitment-v1 entries.

- `schema/content_commitment.mutation-audit.yaml` registered with the Group 4 audit runner. Four mutation switches, each with a binding test.

**Verification for Group 2.** Content commitment primitive is shipped with serialization, verification, building, and lookup. Mutation-audit registry passes. Golden vectors locked. `go test ./schema/... ./builder/...` green for content commitment paths.

## Group 3 — Manifest Escrow Lifecycle Primitive

**Packages:** `crypto/escrow/`, `core/vss/` (read-only reference)

**Purpose:** Ship the lifecycle wrapper that composes v7.75's `SplitV2` with AES-GCM payload encryption to support manifest payloads larger than 32 bytes. This primitive is required by Group 5's archival manifest implementation. It is a thin composition over existing v7.75 primitives — no new cryptography — but it must be well-tested because it is the load-bearing escrow operation for archival records.

**Subgroup 3.1 — `SplitManifest` function.** New function in a new file `crypto/escrow/manifest_escrow.go`:

```go
func SplitManifest(
    manifest []byte,
    M, N int,
    dealerDID string,
) (
    shares []Share,
    manifestCiphertext []byte,
    manifestNonce [12]byte,
    splitID [32]byte,
    commitments []vss.Commitment,
    err error,
)
```

Implementation:

1. Validates `M >= 2`, `M <= N`, `N <= 255`, `dealerDID != ""`.
2. Generates a fresh 32-byte AES-GCM key via `crypto/rand`.
3. Generates a fresh 12-byte AES-GCM nonce via `crypto/rand`.
4. Encrypts `manifest` under the AES-GCM key with the nonce, producing `manifestCiphertext`.
5. Generates a fresh 32-byte SplitID nonce via `crypto/rand`.
6. Computes `splitID = ComputeEscrowSplitID(dealerDID, splitIDNonce)` (the v7.75 deterministic SplitID derivation).
7. Calls `SplitV2(manifestKey, M, N, dealerDID, splitIDNonce)` (the v7.75 escrow primitive) to produce shares and commitments.
8. Zeroizes the manifest key from local memory.
9. Returns the shares, ciphertext, AES-GCM nonce, splitID, and commitments.

The function does NOT distribute shares — distribution is an operator-level operation. The function returns the share package; the caller (typically a domain lifecycle function in Group 5) handles distribution via existing `escrow.EncryptForNode` ECIES wrapping per recipient.

The two distinct nonces — the AES-GCM nonce (12 bytes) and the SplitID nonce (32 bytes) — are independent and must not be confused. The AES-GCM nonce is part of the manifest ciphertext metadata; the SplitID nonce is part of the on-log split commitment per ADR-005. Both are returned to the caller; both must be preserved through the lifecycle.

**Subgroup 3.2 — `ReconstructManifest` function.** New function in `crypto/escrow/manifest_escrow.go`:

```go
func ReconstructManifest(
    shares []Share,
    manifestCiphertext []byte,
    manifestNonce [12]byte,
) (manifest []byte, err error)
```

Implementation:

1. Validates `len(shares) >= 2` (below-threshold check is enforced by `Reconstruct`; this is the SDK-boundary fail-fast).
2. Calls `Reconstruct(shares)` (the v7.75 escrow primitive) to derive the manifest key.
3. Defers a zeroization of the manifest key.
4. Decrypts `manifestCiphertext` under the manifest key with `manifestNonce` via AES-GCM.
5. Returns the plaintext manifest.

If decryption fails (e.g., wrong nonce, corrupted ciphertext, threshold-met but key reconstruction returned the wrong value), the function returns an error wrapping `escrow.ErrManifestDecryptionFailed` — a new sentinel error. The caller handles this as a structural failure, not a threshold failure.

**Subgroup 3.3 — Atomic emission integration.** The composition discipline from ADR-005 §4 extends to manifest escrow: a function calling `SplitManifest` produces a structurally-bound result containing all five outputs (shares, ciphertext, nonce, splitID, commitments). The result type makes "extracting shares without the ciphertext" or "extracting the SplitID without the commitments" structurally impossible:

```go
type ManifestSplitResult struct {
    shares             []Share
    manifestCiphertext []byte
    manifestNonce      [12]byte
    splitID            [32]byte
    commitments        []vss.Commitment
}

func (r *ManifestSplitResult) Shares() []Share
func (r *ManifestSplitResult) Distribution() []ShareDistributionPackage
// etc.
```

The `Distribution` method packages each share with its share-recipient-specific ECIES wrapping context, returning a slice of distribution packages ready for `escrow.EncryptForNode` invocation. This makes the distribution protocol self-describing: a caller iterates the distribution slice and ships each package to its destination operator.

**Closure proof.**

- `TestSplitManifest_HappyPath` splits a 4 KB manifest with M=3, N=5, validates the share count, validates the ciphertext is well-formed AES-GCM (decryption with correct key produces the original manifest), validates the SplitID matches `ComputeEscrowSplitID(dealerDID, splitIDNonce)`.

- `TestReconstructManifest_HappyPath` reconstructs from M shares, validates the result equals the original manifest.

- `TestReconstructManifest_BelowThreshold_Rejected` attempts reconstruction with M-1 shares, validates rejection.

- `TestReconstructManifest_WrongCiphertext_Rejected` reconstructs with correct shares but tampered ciphertext, validates `ErrManifestDecryptionFailed`.

- `TestReconstructManifest_WrongNonce_Rejected` reconstructs with correct shares and ciphertext but wrong nonce, validates rejection.

- `TestSplitManifest_LargeManifest` splits a 64 KB manifest, validates the round-trip succeeds.

- `TestSplitManifest_GoldenVector` pins a fixture at `crypto/escrow/testdata/manifest_split_vector.json` with a known manifest, dealer DID, M, N, and pinned outputs (under deterministic randomness using a test-only `io.Reader`).

- `TestManifestSplitResult_StructuralIntegrity` validates the result type cannot be partially constructed — every successful return populates all five fields.

- `crypto/escrow/manifest_escrow.mutation-audit.yaml` registered. The mutation switches in this group are: `muEnableManifestKeyZeroization` (gates the key zeroization step in `SplitManifest`), `muEnableManifestDecryptionFailedSentinel` (gates the structured error wrapping in `ReconstructManifest`).

**Verification for Group 3.** Manifest escrow primitive is shipped. Round-trip tests pass for various manifest sizes. Mutation-audit registry passes. Golden vector locked. `go test ./crypto/escrow/...` green for manifest escrow paths.

## Group 4 — Authorization Grant Primitive

**Packages:** `schema/`, `lifecycle/`, `verifier/`

**Purpose:** Ship the protocol-level authorization grant primitive. This primitive is required by Group 6's replica issuance lifecycle and by domain applications that need to record authorization decisions on-log with explicit expiration semantics. The schema accommodates both grant semantics (`authorized_party` non-empty: this party may perform the operation) and seal semantics (`authorized_party` empty: this record is sealed; no party currently authorized).

**Subgroup 4.1 — `AuthorizationGrant` struct and serializers.** New types in a new file `schema/authorization_grant.go`:

```go
type AuthorizationGrant struct {
    SubjectPointer   types.LogPosition
    AuthorizedParty  []byte         // empty for seal semantics; otherwise length-prefixed public key
    Expiration       int64          // microseconds since Unix epoch
    PurposeCode      string         // domain-defined categorical
    AuthorityPointer types.LogPosition
    IdentityClaim    []byte         // length-prefixed opaque domain-produced identifier
}
```

`SerializeAuthorizationGrant` produces a length-prefixed payload: each field length-prefixed per the universal v7.75 rule. `DeserializeAuthorizationGrant` validates: subject pointer well-formed, authorized party either empty or a valid public key (33-byte compressed secp256k1 by default; longer for alternative key formats), expiration is a valid int64, purpose code is valid UTF-8, authority pointer well-formed, identity claim is bytes (no length restriction).

The empty-authorized-party case is structurally distinguishable from the non-empty case via the length prefix: a 2-byte big-endian length of 0 followed by zero bytes. The verifier and lifecycle layer dispatch on this distinction.

**Subgroup 4.2 — `authorization-grant-v1` schema registration.** New file `schema/authorization_grant_v1.go` declares the schema, parallel to Group 2's content-commitment-v1 schema registration.

**Subgroup 4.3 — `BuildAuthorizationGrantEntry` builder.** New function in `builder/entry_builders.go`:

```go
type AuthorizationGrantParams struct {
    Destination      string
    SignerDID        string
    Fetcher          types.EntryFetcher
    Grant            AuthorizationGrant
    EventTime        int64
}

func BuildAuthorizationGrantEntry(
    params AuthorizationGrantParams,
) (*envelope.Entry, error)
```

Calls `validateCommon`. Verifies the grant is well-formed via `VerifyAuthorizationGrant`. Verifies the signer's authority over the scope at `Grant.AuthorityPointer` — the signer must be in the scope's authority set per existing v7.75 path classification. Computes the schema fingerprint for the authorization-grant-v1 schema. Serializes the grant and constructs a Path A commentary entry. Returns the unsigned entry.

**Subgroup 4.4 — `VerifyAuthorizationGrant` and `VerifyAuthorizationExpiration`.** Two verifier functions in `schema/authorization_grant.go`.

`VerifyAuthorizationGrant` checks structural validity, gated by mutation switches:

1. `muEnableAuthorizationSubjectPointerNonZero` — `SubjectPointer` is well-formed.
2. `muEnableAuthorizationAuthorityPointerNonZero` — `AuthorityPointer` is well-formed.
3. `muEnableAuthorizationExpirationFuture` — `Expiration > 0` (the value is not the zero default; an explicit expiration is required).

`VerifyAuthorizationExpiration` is called by downstream verifiers that reference an authorization grant. Signature:

```go
func VerifyAuthorizationExpiration(
    grant *AuthorizationGrant,
    referencingEventTime int64,
) error
```

Validates `referencingEventTime <= grant.Expiration`. The mutation switch `muEnableAuthorizationExpirationEnforced` gates this check — flipping it false allows downstream entries to reference expired authorizations, which is the failure mode the gate prevents.

**Subgroup 4.5 — `FetchAuthorizationGrant` lookup primitive.** New function:

```go
func FetchAuthorizationGrant(
    fetcher types.EntryFetcher,
    pointer types.LogPosition,
) (*AuthorizationGrant, error)
```

Parallel structure to Group 2's `FetchContentCommitment`.

**Closure proof.**

- `TestAuthorizationGrant_RoundTrip` for grant semantics (non-empty authorized party) and seal semantics (empty authorized party).

- `TestAuthorizationGrant_GoldenVector` pins a fixture at `schema/testdata/authorization_grant_vector.json`.

- Three binding tests for the structural mutation switches.

- `TestVerifyAuthorizationExpiration_HappyPath` validates a referencing entry within the expiration window passes.

- `TestVerifyAuthorizationExpiration_Expired_Rejected` validates a referencing entry past the expiration is rejected.

- `TestBuildAuthorizationGrantEntry_HappyPath` validates a complete build round-trip.

- `TestBuildAuthorizationGrantEntry_SignerNotInAuthority_Rejected` validates rejection when the signer is not in the scope's authority set.

- `TestFetchAuthorizationGrant_HappyPath` and `_WrongSchema_Rejected` parallel to Group 2.

- `schema/authorization_grant.mutation-audit.yaml` registered with four mutation switches and binding tests for each.

**Verification for Group 4.** Authorization grant primitive is shipped. Both grant and seal semantics are supported and tested. Expiration enforcement is gated and binding-tested. Mutation-audit registry passes. Golden vector locked.

## Group 5 — Archival Manifest Lifecycle

**Packages:** `lifecycle/`, `crypto/escrow/` (read-only reference for Group 3 primitives), `schema/` (read-only reference for Groups 2 and 4 primitives)

**Purpose:** Ship the high-level lifecycle function that orchestrates Groups 1-4 into the complete archival record provisioning flow. This is where v7.76 becomes actually usable for domain applications. A domain caller invoking `BuildArchivalRecord` provides plaintext record bytes, a record-content schema reference, escrow node target identifiers, and the (M, N) threshold. The function produces three on-log entries plus the share distribution package, atomically.

**Subgroup 5.1 — `archival-manifest-v1` schema definition and registration.** New file `schema/archival_manifest_v1.go`. Parallel structure to the other schema registrations. The schema's domain payload is the serialized `ArchivalManifest` struct:

```go
type ArchivalManifest struct {
    RecordIdentifier          string
    SchemaRef                 types.LogPosition
    SchemaFingerprint         [32]byte
    ManifestSplitID           [32]byte
    ManifestCommitmentPointer types.LogPosition
    MasterCommitmentPointer   types.LogPosition
}
```

Serialization, deserialization, builder (`BuildArchivalManifestEntry`), verifier (`VerifyArchivalManifest`), and lookup (`FetchArchivalManifest`) follow the patterns established in Groups 2 and 4.

**Subgroup 5.2 — `BuildArchivalRecord` lifecycle function.** New function in a new file `lifecycle/archival_record.go`:

```go
type ArchivalRecordParams struct {
    Destination       string
    SignerDID         string
    Fetcher           types.EntryFetcher
    RecordIdentifier  string
    RecordSchemaRef   types.LogPosition  // domain's record-content schema (e.g., birth-certificate-record-v1)
    RecordPlaintext   []byte
    DealerDID         string
    M                 int
    N                 int
    EventTime         int64
}

type ArchivalRecordResult struct {
    archivalManifestEntry         *envelope.Entry
    masterContentCommitmentEntry  *envelope.Entry
    escrowSplitCommitmentEntry    *envelope.Entry
    shareDistribution             []ShareDistributionPackage
}

func BuildArchivalRecord(
    params ArchivalRecordParams,
) (*ArchivalRecordResult, error)
```

The function orchestrates:

1. Validates parameters: destination valid, signer DID non-empty, fetcher non-nil, record identifier non-empty, record schema ref non-zero, record plaintext non-empty, dealer DID non-empty, `2 <= M <= N <= 255`.

2. Computes the record schema's fingerprint via `ComputeSchemaFingerprint(fetcher.Fetch(RecordSchemaRef))`.

3. Serializes the record plaintext under the record's schema (the schema's canonical serialization rules, declared by the domain — typically JSON with field-order discipline). This produces the `serialized_record_bytes` that the master commitment will commit to.

4. Computes the master content commitment digest via `ComputeContentCommitmentDigest(RecordSchemaRef, serialized_record_bytes)`.

5. Builds the master content-commitment-v1 entry via `BuildContentCommitmentEntry` with `Purpose: PurposeMaster`. The subject pointer for the master commitment is the (yet-to-be-determined) log position of the archival manifest entry — this circular dependency is handled by atomic emission: the three entries are submitted to the log in a single batch, and the operator's batch admission resolves the cross-references.

6. Calls `escrow.SplitManifest(serialized_record_bytes, M, N, DealerDID)` from Group 3 to produce the share package. The serialized record bytes are the manifest in this design — they ARE the off-log content that escrow protects.

7. Builds the escrow-split-commitment-v1 entry (existing v7.75 primitive) using the commitments and SplitID from `SplitManifest`.

8. Builds the archival-manifest-v1 entry referencing the master content commitment (Step 5) and the escrow split commitment (Step 7) via their (yet-to-be-determined) log positions, and pinning the manifest's SplitID and the record schema's fingerprint.

9. Bundles the three entries plus the share distribution package into an `ArchivalRecordResult`. Returns.

The atomic emission discipline: the three entries reference each other via log positions that don't yet exist (since they're being produced together). The result type forces the caller to submit all three entries in a single batch to the operator's admission API; the operator processes the batch atomically, resolving cross-references after all three positions are known. The `muEnableArchivalRecordAtomicEmission` mutation switch gates an internal assertion that all four result fields (three entries plus shares) are non-nil before return.

**Subgroup 5.3 — Operator-side admission integration.** This is operator-facing work, but the SDK provides the validation primitives. New function in `core/envelope/admission.go`:

```go
func ValidateArchivalRecordBatch(
    batch []*envelope.Entry,
    fetcher types.EntryFetcher,
) error
```

Validates that a batch of entries forms a complete archival record submission: exactly one archival-manifest-v1, exactly one content-commitment-v1 with `Purpose: PurposeMaster`, exactly one escrow-split-commitment-v1; the cross-pointers correctly resolve once the batch's log positions are assigned; the schema fingerprints match across all three entries.

The operator's admission pipeline calls this function before assigning log positions to a batch. If validation fails, the entire batch is rejected.

## Group 6 — Replica Issuance Lifecycle and Derivation Binding

**Packages:** `lifecycle/`, `schema/`, `crypto/signatures/` (read-only reference)

**Purpose:** Ship the replica issuance lifecycle that produces transparent, verifiable replicas of archival records. This is the "transparency" property of v7.76: a domain caller authorizing a replica issuance produces an on-log chain that any external party can verify against the master record.

**Subgroup 6.1 — `DerivationBinding` struct and serializers.** New types in a new file `schema/derivation_binding.go`:

```go
type DerivationBinding struct {
    MasterCommitmentPointer  types.LogPosition
    DerivedCommitmentPointer types.LogPosition
    BindingProof             BindingProof
}

type BindingProof struct {
    Algorithm uint8 // 0x01 = Schnorr secp256k1 via LengthPrefixed
    ProofBytes []byte // length depends on algorithm; 64 bytes for Schnorr secp256k1
}
```

Serialization, deserialization, schema registration, and lookup follow the patterns from Groups 2 and 4.

**Subgroup 6.2 — `ComputeBindingProofMessage` and `SignBindingProof` primitives.** New functions in `schema/derivation_binding.go`:

```go
func ComputeBindingProofMessage(
    masterCommitmentValue, derivedCommitmentValue []byte,
    schemaRef types.LogPosition,
    schemaFingerprint [32]byte,
) [32]byte

func SignBindingProof(
    message [32]byte,
    signerKey []byte, // 32-byte secp256k1 private key
) (BindingProof, error)
```

`ComputeBindingProofMessage` calls `LengthPrefixed("ORTHOLOG-V7.76-DERIVATION-BINDING-v1", masterCommitmentValue, derivedCommitmentValue, schemaRef.CanonicalBytes(), schemaFingerprint[:])`. The DST is versioned per ADR-006 §3.

`SignBindingProof` produces a Schnorr signature over the message using the signer's secp256k1 key. The signature is 64 bytes (32-byte R + 32-byte S). Returns a `BindingProof` with `Algorithm: 0x01` and `ProofBytes` containing the signature.

**Subgroup 6.3 — `VerifyDerivationBinding` verifier.** New function:

```go
func VerifyDerivationBinding(
    binding *DerivationBinding,
    masterCommitment, derivedCommitment *ContentCommitment,
    signerPublicKey []byte,
) error
```

Verifies:

1. `muEnableDerivationBindingMasterPointerNonZero` — master pointer well-formed.
2. `muEnableDerivationBindingDerivedPointerNonZero` — derived pointer well-formed.
3. `muEnableDerivationBindingAlgorithmRecognized` — the proof algorithm discriminator is recognized (0x01 for v7.76).
4. `muEnableDerivationBindingSchemaRefMatch` — both commitments reference the same schema.
5. `muEnableDerivationBindingSchemaFingerprintMatch` — both commitments pin the same schema fingerprint.
6. `muEnableDerivationBindingSignatureValid` — the binding proof signature verifies against the signer's public key over the computed message.

The five mutation switches gate the five verification properties. Each has a binding test that constructs a binding violating exactly that property and confirms rejection.

**Subgroup 6.4 — `BuildReplicaIssuance` lifecycle function.** New function in a new file `lifecycle/replica_issuance.go`:

```go
type ReplicaIssuanceParams struct {
    Destination               string
    SignerDID                 string
    Fetcher                   types.EntryFetcher
    MasterArchivalManifestPtr types.LogPosition
    AuthorizationPointer      types.LogPosition
    RecipientPublicKey        []byte
    RecorderInstitutionalKey  []byte // 32-byte private key for binding signature
    EventTime                 int64
}

type ReplicaIssuanceResult struct {
    derivedContentCommitmentEntry *envelope.Entry
    derivationBindingEntry        *envelope.Entry
    deliveryAuthorizationEntry    *envelope.Entry
    replicaCiphertext             []byte
    wrappedReplicaKey             []byte
}

func BuildReplicaIssuance(
    params ReplicaIssuanceParams,
) (*ReplicaIssuanceResult, error)
```

The function orchestrates:

1. Validates parameters.

2. Fetches the master archival manifest entry.

3. Fetches the master content commitment entry (via the manifest's `MasterCommitmentPointer`).

4. Fetches the authorization grant entry (via `AuthorizationPointer`).

5. Verifies the authorization grant: signed by an authority-set member, unexpired relative to `EventTime`, references the master archival manifest via subject pointer, names the recipient via `AuthorizedParty`.

6. Reconstructs the master plaintext via M-of-N share collection plus `escrow.ReconstructManifest`. (Note: this step requires operator-coordinated escrow share collection; the SDK function takes the collected shares as an additional input parameter omitted above for brevity. In practice, the recorder's enclave collects shares and invokes this function with shares in scope.)

7. Generates a fresh AES-GCM key for the replica. Encrypts the master plaintext under the fresh key. The ciphertext is the replica.

8. Wraps the fresh key under the recipient's public key via ECIES (`escrow.EncryptForNode`).

9. Computes the derived content commitment digest via `ComputeContentCommitmentDigest(RecordSchemaRef, replicaCiphertext)`.

10. Builds the derived content-commitment-v1 entry with `Purpose: PurposeDerived`, subject pointer = master archival manifest, schema ref = record schema ref, fingerprint pinned.

11. Computes the binding proof message and signs it with the recorder's institutional key. Builds the derivation-binding-v1 entry.

12. Builds a delivery-record authorization-grant-v1 entry recording the actual issuance (subject = derived commitment, authorized party = recipient, expiration = far future since the replica is now persistent, purpose = "delivery-record", authority pointer matching the authorizing scope).

13. Bundles the three entries plus the replica ciphertext and wrapped key into a `ReplicaIssuanceResult`.

14. Zeroizes the master plaintext and the replica AES-GCM key.

The atomic emission discipline: all three entries plus the ciphertext and wrapped key are produced atomically. The mutation switch `muEnableReplicaIssuanceAtomicEmission` gates the assertion.

**Subgroup 6.5 — `VerifyReplicaChain` audit-time verifier.** New function:

```go
func VerifyReplicaChain(
    masterArchivalManifest *envelope.Entry,
    masterContentCommitment *envelope.Entry,
    authorizationGrant *envelope.Entry,
    derivedContentCommitment *envelope.Entry,
    derivationBinding *envelope.Entry,
    deliveryAuthorization *envelope.Entry,
    fetcher types.EntryFetcher,
) error
```

Verifies the complete chain structure from on-log data:

1. Master manifest's `MasterCommitmentPointer` resolves to master content commitment.
2. Authorization grant's subject pointer resolves to master manifest, signer is in authority set, unexpired relative to derivation binding's event time.
3. Derived content commitment has `Purpose: PurposeDerived`, schema ref matches master, fingerprint matches master.
4. Derivation binding's master pointer resolves to master content commitment, derived pointer resolves to derived content commitment, schemas match, fingerprints match, binding proof signature verifies.
5. Delivery authorization references derived commitment, signer is in authority set, authorized party matches the original authorization's authorized party.

This is the audit-time function any external auditor calls to verify a replica chain. Six gates, one per property, each binding-tested.

## Boundaries: SDK vs Operator vs Store

This plan is exclusively SDK-side work. For clarity on what falls outside:

**Operator-side work (not in this plan):** Each operator extends their admission pipeline to call `ValidateArchivalRecordBatch` for v7.76 entry batches. Operators that participate in archival-record provisioning coordinate share distribution to escrow nodes via the existing `escrow.EncryptForNode` ECIES wrapping plus their own delivery channels (HTTPS endpoints, mailed HSM cards, etc.). Operators serving archival-record reads coordinate share collection from M-of-N escrow nodes and pass shares to the recorder's `BuildReplicaIssuance` invocation. None of this requires SDK changes; the SDK provides the primitives and the validation functions.

**Store-side work (not in this plan):** Storage of the new entry types uses the existing v7.75 entry storage schema. No new tables, no new indexes specifically for v7.76. Optional domain-specific indexing of `archival-manifest-v1` entries by `record_identifier` is operator policy, implemented against the existing entry table.

**External monitoring services (not in this plan):** Metering, billing, customer registries, audit analytics, alerting, retrieval-proof verification — none of these are SDK concerns. External services consume the public log via existing query primitives and compute whatever they need.

The SDK ships the cryptographic and protocol substrate. Operators and external services compose on top of it using documented interfaces. v7.76 is the SDK contribution; the operational layer above it is domain-driven.
